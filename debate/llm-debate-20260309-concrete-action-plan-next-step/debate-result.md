## Codex (OpenAI GPT-5.2) の批評
### 結論（最初に明言）
Claude案は「壊れた計測器で得たデータを、サンプリングと再学習で救う」という発想であり、根本原因（計測器の破損）を修理していない。したがってこの順序で進めるのは誤りである。次にやるべきことは **計測器（アノテーション仕様・ラベリング手順・グラフ生成）の修理と検証** であり、そこで否定が確定した時点で **Phase 3/4を打ち切る** のが最短である。

### 批判的分析
**1) 論理的整合性**
- Phase 3 が「失敗/判定不能」になった原因は「測定の問題」である、と共有コンテキストで確定している。にもかかわらず Claude案は「サンプルを整える→分類器を再学習→再検定」であり、測定の破綻（ラベル定義・付与規則・グラフ生成の整合性）を直接修正していない。原因に刺さらない手順は無駄である。
- 「各トライグラム×各フィールドで最小10件」という制約は、Phase 3 の対象が **遷移グラフ（ノードとエッジ）** である以上、目的関数がズレている。必要なのはトライグラムの周辺分布ではなく、**64ノードの活性化と遷移エッジの十分な支持数** である。

**2) 実現可能性（隠れコスト込み）**
- 400件程度のゴールドセットは小さい。64クラスを扱う時点で、1クラスあたり平均6件台に落ちる。これで「分類器を信頼できる」水準の誤差評価は成立しない。成立するのは「失敗していることの確認」だけである。
- TF‑IDF が「トライグラム被覆が悪い」という事実は、データバランス以前に **表現が不適切** であることを意味する。バランスを整えても表現が死んでいる限り性能は出ない。再学習に工数を投下する順序は誤りである。
- LLM再注釈の PoC で 100件から 37/64 を出せた点は前進だが、ここから 64/64 を安定化するには「手順書（ルーブリック）・二重注釈・不一致解決」の運用が必須であり、そこを省くと低品質注釈が再生産される。

**3) 新規性の真偽**
- 「層化サンプリングでゴールドを作る」「バランス化して再学習」は標準手法であり新規性はない。新規性を立てるなら、**計測器の妥当性（validity）を定義し、破綻を定量化し、修理プロトコルを提示する** 方向に寄せる必要がある。

**4) スケーラビリティ**
- 計測器が壊れたまま 20,000 件へ増やす行為は、誤差を拡大して計算資源とコストを燃やすだけである。スケールさせる対象はデータ量ではなく、**ラベリングの信頼性と自動化パイプライン** である。
- 「全 22,000 LLM calls は高い」なら、全件再注釈ではなく **能動学習（不確実・欠損クラス・遷移の薄い部分を優先）** に切り替えるのが合理である。

**5) 反例・エッジケース**
- 34/64 しか出ない現状は「データに存在しない」のではなく、「計測器が特定クラスしか吐けない」状態で説明できる。これを放置すると、グラフは構造的に疎になり、Test C の z が“良い結果”に見えるアーティファクトが再発する。
- 64分類が本質的に単一ラベルで定義できない（多義・文脈依存・複数卦の同時成立）場合、単一ラベル分類と単純な遷移グラフはモデル化として誤りである。この前提検証を先にやらない限り、いくら再学習しても収束しない。

**6) 代替案との比較**
- Claude案は「既存TF‑IDF分類器を中心」に据え過ぎている。正しい中心は **測定の妥当性** である。分類器は手段であり、（必要なら）埋め込み＋軽量分類、あるいは LLM 直接分類＋監査の方が早い。

  - 「2–3 sessions」という工数見積は甘い。計測器修理（仕様・監査・一致率評価）を入れた時点で最低でも数日単位になる。ここを削る案は再現不能な結果を生むだけである。

### より正確な見解
以下は **優先順位付き・具体手順・工数・Go/No‑Go** を含む最短ルートである（LLMコール数でコスト上限を管理する）。

#### Step 0（半日）: “計測器”を定義して受入条件を固定
- **やること**：ラベル定義（64卦が単一ラベルか/多ラベルか）、遷移の定義（同一ケース内か時系列か）、グラフ生成仕様（ノード/エッジの作り方）を1枚に固定。  
- **受入条件（この条件を満たさない限り Phase 3 を回さない）**：  
  - 64ノードのうち **≥60 が非ゼロ頻度**  
  - 各ノードの **最小支持数 ≥5**（ゴールド上）  
  - 主要指標（例：次数分布・連結性）が仕様通り再現  
- **No‑Go**：単一ラベル定義が破綻する（同一入力に複数卦が妥当、が常態）なら、単一ラベル分類＋遷移グラフ仮説を終了し、表現を「分布（確率）グラフ」に作り替える。

#### Step 1（1–2日、LLM 800–1,500 calls）: 64ノードを必ず出す “金の最小セット” を作る
- **やること**：  
  - 既存注釈＋現分類器で候補を引き、**欠損している卦を埋める目的**で能動的にサンプルを集める（ランダム層化ではない）。  
  - 目標：**640件**（各卦10件）＋遷移が薄い箇所を追加で**+160件**。  
  - LLMは **二重注釈（独立2回）** を必須にし、不一致のみ第三者（別プロンプト/別モデル）で裁定。  
- **判定指標**：卦ごとの再現率、二重注釈一致率（クラス別に出す）。  
- **No‑Go**：二重注釈一致率が低く、裁定しても安定しない卦が多数なら、卦ラベル自体が測定不能である。Phase 3/4 を中止し、「注釈不安定性の分析＋方法論」に成果を切り替える。

#### Step 2（1日）: “TF‑IDF中心”を捨て、ラベラーを作り直す（最小コスト運用）
- **やること**：  
  - TF‑IDFを主系から外し、埋め込み＋軽量分類（または LLM 直接分類）を主にする。  
  - 運用は **ハイブリッド**：自動ラベルは高信頼のみ採用、低信頼は LLM に回しゴールドへ追加（能動学習ループ）。  
- **Go/No‑Go**：ゴールドに対して **macro‑F1 ≥0.75 かつ 欠損卦の再現率が基準超え**。達成できないなら「グラフ構築の自動化」は失敗であり、全件分析を諦めてサブセット研究に縮退する。

#### Step 3（半日）: グラフを“検定可能”な密度にする
- **やること**：高信頼ラベルだけで遷移グラフを再構築し、疎性を制御（閾値を明示）。  
- **Go条件**：アクティブノード **≥60**、平均次数が極端に小さくない（孤立が例外）。  
- **No‑Go**：ここで 17/64 に戻るなら、Phase 3 の入力が成立していない。検定は中止が正しい。

#### Step 4（1日）: Phase 3 を “感度分析込み” で再実行し、ここで打ち切り判断
- **やること**：  
  - 複数の閾値（高信頼のみ→中信頼含む）でグラフを作り、結果が閾値で崩れないことを確認。  
  - 疎グラフ由来のアーティファクト（Test C型）を排除するため、疎性と次数を保持したヌルで比較する。  
- **No‑Go（打ち切り条件）**：A/B が再現的に失敗し、感度分析でも改善しないなら、同型仮説は棄却である。Phase 4 へ進む行為は研究倫理として誤りである。

#### Step 5（1–2日）: “否定結果でも出せる成果” に即座に切り替える
- **やること**：  
  - 計測器修理のプロトコル、注釈一致率、疎性が統計を壊す実証（Test Cがアーティファクトになる条件）をまとめる。  
  - 「同型は成立しない」を主張し、代わりに成立する構造（局所モチーフ、部分同型、クラスタ）へ主題を移す。  
- **効果**：失敗を“再現可能な否定結果”として出版可能な形に変換する。

---

この順序で進めれば、次の2点が同時に達成できる：  
(1) Phase 3 が「検定不能」から「検定可能」へ移る、(2) 否定が出た瞬間に止められる。  
必要なら、Step 0 の「受入条件」をあなたの Phase 3 指標（A/B/C）に合わせて数値まで固定して、完全な停止ルールに落とし込む。
