# LLM Debate: 診断質問のUX改善

## 議題
診断質問のUX改善

## Claude (Anthropic) の見解
現在の質問は内部分類軸に寄りすぎている。改善案：(1)具体的なシナリオベースの質問に変える（例：「今、転職を考えていますか？」→「最近、新しいことを始めたいと感じることが多い」Yes/No）(2)ストーリー形式で状況を選ばせる（例：「あなたの今の状況に近いのはどれ？」→3つの物語から選択）(3)段階的開示：最初の2問で大まかな方向を掴み、残りは絞り込み用に動的生成。課題：質問を具体化するほど適用範囲が狭まり、ユーザーが「どれも当てはまらない」と感じるリスクがある。

## Codex (OpenAI GPT-5.2) の批評
### 結論（最初に明言）
Claudeの改善案は「ユーザーが答えやすい形にする」という方向性だけは正しいが、中心提案が粗雑である。質問をシナリオ化・ストーリー化するだけでは診断の再現性が崩れ、占い卒業層が求める「納得できる根拠」をむしろ失う。正解は、内部の分類軸（Q1〜Q5）を捨てることではなく、**軸をユーザー言語の行動指標に翻訳し、混合状態・不確実性を許容する測定設計に置き換えること**である。

### 批判的分析
1. **論理的整合性**
   - 現状の問題は「抽象的だから」ではなく、**内部軸→ユーザー回答への写像が欠落している**点である。ユーザーは自分の状況を“拡大/縮小/維持/転換”に直接マッピングできない。結果として回答は当て推量になり、診断の入力がノイズ化する。
   - Claude案は「内部軸に寄りすぎ」を批判しつつ、**新しい質問が内部軸を正しく測る保証**を提示していない。UX改善の顔をしたモデル破壊である。

2. **実現可能性（隠れコストを含む）**
   - シナリオ文面の差し替え自体は実装容易だが、価値を決めるのはコードではなく**設問の品質管理（文の一義性、偏り、文化依存、二重質問の排除、回答分布の監視）**である。ここを抜くと改善ではなく劣化になる。
   - 「残りを動的生成」は実務コストが高い。生成の正しさを担保するには、**設問バンク・分岐ロジック・ログ設計・品質検証**が要る。これを省略すると“それっぽい質問を出す装置”になり、占い感が増幅する。

3. **新規性の真偽**
   - シナリオ質問、ストーリー選択、段階的開示は既存の定番手法であり、新規性はない。新規性を語るなら、**測定の安定性（再テスト一致）と説明可能性（なぜその結果か）**まで含めて差分を示す必要がある。

4. **スケーラビリティ**
   - ストーリー方式はスケールしない。対象（転職/部署異動/独立/復職/介護など）が増えるほど、**「どれも当てはまらない」問題が構造的に増える**。増設はコンテンツ負債として積み上がる。
   - 動的分岐はスケールするが、前提は「同じ軸を同じ意味で測り続ける」ことだ。設問が都度変わる設計は、**時系列比較・改善検証・説明責任**を難しくする。

5. **反例・エッジケース**
   - キャリア転換期は混合状態が標準である（例：方向転換したいが当面は維持、主体的に動くが環境要因が支配的、前向きと不安が同居）。単一選択・Yes/Noはこの現実を切り捨て、誤分類を量産する。
   - 「占い卒業」層は、質問が心理テスト風でも根拠が見えないと拒否する。ストーリーは“演出”に見えやすく、信頼を落とす。

6. **代替案との比較**
   - より優れたアプローチは存在する。**内部軸を維持しつつ、行動・制約・意思決定の言葉で測る**設計（＋混合回答の許容、＋説明可能なスコア化）が、UXと信頼性を同時に上げる。

### Claudeの意見への反論/修正
- (1) **シナリオベースYes/No化は誤りである。**情報量を削り、曖昧文への同意癖（同意バイアス）でスコアが歪む。必要なのは二値化ではなく、**「どの程度近いか」か「どちらにより近いか（強制選択）」**である。  
- (2) **ストーリー選択は測定の代替にならない。**オンボーディングの“入口”としては有効だが、診断の“本体”に据えると網羅性が破綻する。使うなら**文脈選択（例：主に仕事/対人/家庭）**に限定し、本体は軸を測る設問で行うべきである。  
- (3) **段階的開示は正しいが、「動的生成」は雑である。**正解は生成ではなく、**事前に設計した設問バンクから適応的に出題**することだ。これなら説明可能性と品質管理を維持できる。  
- Claudeが挙げた「具体化すると適用範囲が狭まる」は単なる懸念ではない。**具体化の失敗は必ず起きる**。回避策（“最も近い”選択、複数選択、自由記述フォロー）が設計に含まれていない時点で提案として未完成である。

### より正確な見解
- **本質**：UX課題は「抽象度」ではなく「ユーザーの自己記述（状況・行動・制約）を、診断軸へ誤差少なく変換できていない」点である。ここを直さずに演出だけ変えると、占い感だけが残る。
- **処方箋（最短で効く設計）**
  1. Q1〜Q5を維持し、各軸を**行動指標の設問**に分解する（例：転換=「役割/業界/働き方を変える意思決定が中心」、縮小=「責任/負荷を減らす決断が中心」）。  
  2. 回答形式を**単一カテゴリから混合許容へ**変更する（例：感情は“最も強い”＋“次に強い”、主体性は“自分で動ける度合い”の段階尺度）。  
  3. **段階的開示は「高判別の2問→不足軸だけ追加」**で行う（出題は設問バンクから選択）。質問数削減と精度を両立できる。  
  4. 「占い卒業」対策として、結果画面で**回答→結論の対応（なぜそう出たか）**を短く明示する。説明不能な診断は信用されない。
- **成功条件**：完了率ではなく、**自己納得（当てはまり評価）と再テスト一致**をKPIに置く。ここを測らない改善は“見た目の改善”で終わる。


## 統合分析

### 共通点
1. **現状の問題認識**: 現在の質問（Q1〜Q5）は抽象的すぎて、ユーザーが自分の状況をマッピングできない
2. **UX改善の必要性**: ユーザーが答えやすい形への変換が必要
3. **段階的開示の有効性**: 最初に少数の高判別質問、その後適応的に追加という方向性は一致
4. **「占い卒業」層への配慮**: 根拠が見える設計が信頼獲得に必須

### Codex批評の核心的指摘
Codexの批評は厳しいが正当である。私（Claude）の初期提案は「演出の改善」に偏り、**測定の信頼性**を軽視していた。

特に重要な指摘:
1. **シナリオYes/No化は情報量を削り、同意バイアスでスコアが歪む**
2. **ストーリー選択はスケールしない**（対象が増えると「どれも当てはまらない」問題が構造的に増加）
3. **動的生成は品質管理コストが高い**（設問バンクからの適応出題が正解）
4. **混合状態が標準**（キャリア転換期は複数の軸が同時に動く）

### 結論・推奨

**採用すべき設計方針**:

1. **内部軸（Q1〜Q5）は維持**し、各軸を**行動指標の言葉に翻訳**する
   - 例: 「転換」→「役割/業界/働き方を変える意思決定が中心ですか？」
   - 例: 「縮小」→「責任/負荷を減らす決断を検討していますか？」

2. **回答形式を混合許容へ変更**
   - 単一カテゴリ選択 → 「最も近い」＋「次に近い」
   - 感情軸 → 複数選択可（前向きと不安は共存する）
   - 主体性 → 段階尺度（「自分でコントロールできる度合い」1-5）

3. **設問バンク方式の適応出題**
   - 事前に設計した設問バンク（各軸3-5問）を用意
   - 最初の2問で高判別 → 不確実な軸のみ追加質問
   - 動的生成は使わない（品質管理不能）

4. **説明可能性の担保**
   - 結果画面で「なぜこの結果か」を短く明示
   - 「あなたの回答: 転換傾向+前向き+対外 → 乾為天が候補に」

5. **KPI設計**
   - 完了率だけでなく、**自己納得度**（当てはまり評価）と**再テスト一致率**を測定
   - これを測らない改善は見た目だけの改善
