# LLM Debate: 人生の現在地診断サービスの実装計画

## 議題
人生の現在地診断サービスの実装計画

## Claude (Anthropic) の見解
【既存資産】Flask API(standard/diary/backtrace 3モード), 5軸診断システム(v5仕様凍結), FeedbackEngine(5レイヤー), BacktraceEngine(逆算), LLM構造化抽出, 確率テーブル, 事例検索エンジン, 11,336件DB, Cloudflare Workers API。

【方針要約】ターゲット=順調だが違和感がある人。提供価値=現在地・未来分岐・次の一歩の整理。商品=現在地診断レポート(6セクション: 現在地分析/今のまま進んだ場合/別ルート2-3本/各ルートのリスク・行動/今日やること/30-90日実験プラン)。ポジショニング=占い風演出+実例データ診断。チャネル=ココナラ・LINE。

【提案する実装計画 5フェーズ】

Phase 1: 質問フロー再設計(2週間)
- 既存5軸診断を「違和感の言語化」型に変換
- 問題解決型質問→違和感探索型質問へリライト
- LLM構造化抽出プロンプトを対個人用に最適化
- 出力: 新しい質問フロー仕様書

Phase 2: レポート生成エンジン(3週間)
- 方針の6セクション構造でレポートテンプレート設計
- BacktraceEngineの逆算ロジックを「未来分岐」表現に変換
- FeedbackEngineの5レイヤーを6セクションにマッピング
- LLMによる自然言語レポート生成(占い風演出+データ根拠)
- 出力: report_generator.py + テンプレート

Phase 3: MVP公開(2週間)
- ココナラ出品用のシンプルなフロー構築
- 入力: テキストベース質問(ココナラメッセージ)
- 処理: LLM抽出→診断→レポート生成
- 出力: PDF/テキストレポート
- 手動オペレーション前提(自動化はPhase 4)

Phase 4: LINE連携+自動化(3週間)
- LINE公式アカウント+リッチメニュー
- 質問フローをLINEボット化
- Stripe/PAY.JP決済連携
- Cloudflare Workers APIをフロントエンドとして活用

Phase 5: LP+マーケティング(2週間)
- LP設計・実装(方針のコピー案を使用)
- SEO対策コンテンツ
- ココナラ最適化(タイトル・説明文・価格設定)

【懸念点】
1. individual事例3,217件で「人生の分岐」診断に十分か
2. 5軸診断→違和感言語化の変換が自然にできるか
3. ココナラの手動運用とLINE自動化の並行はリソース的に現実的か
4. 占い風演出とデータ診断のバランスをどう取るか
5. 価格設定(ココナラ相場vs提供価値)

## Codex (OpenAI GPT-5.2) の批評
### 結論（最初に明言）
この実装計画は **不採択である**。理由は単純で、「現在地診断」「未来分岐」を名乗るのに必要な **妥当性の定義・検証・運用品質** が設計に存在せず、占い演出で誤魔化す構図になっているため、プロダクトとして **信頼と再現性が成立しない**。

### 批判的分析
#### 1. 論理的整合性
- **「データ診断」×「占い風演出」は両立しない。** データ根拠を掲げた瞬間に説明責任が発生する。占い演出はその説明責任を回避する装置であり、同居させると「都合が悪いときだけ占い」を疑われて信頼が崩壊する。
- **「未来分岐」の根拠が定義されていない。** BacktraceEngineを「未来分岐」表現に変換する発想は、因果と反実仮想を混同している。過去事例の類似検索と逆算ロジックは、未来の分岐を“予測”したことにならない。ここを曖昧にしたまま商品化すると誇大になる。
- **5軸診断の仕様凍結と質問フロー全面改変が矛盾する。** 質問を「違和感言語化」型に変えるなら、5軸スコアの意味は再較正が必須であり、凍結仕様は維持できない。

#### 2. 実現可能性（隠れコスト込み）
- **12週間の工程は見積もりとして破綻している。** 実装そのものより、品質保証・安全設計・運用設計が支配的コストになるのに、計画がそこをゼロ扱いしている。
- **LLMを“推論エンジン”として使う設計は運用品質が出ない。** 構造化抽出→診断→レポート生成の鎖は、どこか一段が揺れるだけで最終出力が崩れる。品質を担保するには評価セット、回帰テスト、ガードレール、失敗時のフォールバックが要る。計画に存在しない。
- **PDF/テキスト納品は地味にコストが高い。** 体裁、誤字、禁則、図表、個人情報の扱い、再生成、差分対応が発生し、手動運用の工数が直撃する。

#### 3. 新規性の真偽
- **新規性は弱い。** 「質問→診断→パーソナライズレポート→行動提案」は既存の自己分析・コーチング・AI相談の定番構造である。差分は「11,336件DB」と「逆算ロジック」だが、そこが“効く”ことを示す設計がないため新規性として機能しない。

#### 4. スケーラビリティ
- **手動×LINE自動化の並行は運用が崩れる。** 初期は例外処理が多発する。例外を抱えたまま自動化すると、バグではなく“顧客トラブル”として表面化する。
- **コスト構造が未定義でスケール不能。** 1レポートあたりのLLMコスト、人的レビュー時間、返金率、問い合わせ対応時間を見積もっていない。これがないスケール議論は空論である。

#### 5. 反例・エッジケース
- **危機ケース（希死念慮、DV、依存、重いメンタル不調）で破綻する。** 「今日やること」を誤って提示すると被害が出る。危機検知・制限・誘導（専門窓口へ）を設計に入れない限り、提供自体が無責任である。
- **入力が薄い／誇張／矛盾するケースで出力が破綻する。** ココナラのテキスト往復は情報欠損が常態であり、構造化抽出の前提が崩れる。
- **“データ根拠”の提示が個人情報と衝突する。** 実例DBを根拠として引用するほど再同定リスクが上がる。匿名化・要約・統計化の設計が必要だが計画にない。

#### 6. 代替案との比較
- **現計画は「見せ方先行」で、代替案に負ける。** 代替案は「診断」ではなく「意思決定支援」として、(a)現状の構造化、(b)選択肢の列挙、(c)小さな実験計画の提示、(d)振り返りテンプレ、を堅牢に提供する。ここに実例DBを“補助証拠”として添える方が、誠実で再現性が出る。

### Claudeの意見への反論/修正
- **フェーズ設計は甘い。** Phase 1〜2で「質問再設計」と「レポート生成」を作れば価値が出る前提が誤りである。価値は出力の体裁ではなく、意思決定が改善する再現性で決まる。再現性を担保する評価設計が抜けている。
- **懸念1（事例数）は論点がズレている。** 必要なのは件数ではなく、事例が「入力特徴→介入→結果」の形で整備されていることだ。整備されていないDBは1万件でも根拠にならない。
- **懸念4（占い×データのバランス）は“バランス”ではない。** これは設計矛盾であり、混ぜるほど信用が落ちる。片方を主、片方を従に固定し、説明責任の境界を明文化しなければ成立しない。
- **良い点もある。** ターゲット設定（順調だが違和感）は明確で、提供物（6セクション）も購買イメージを作れる。だがそれは「売り方」の整備であり、「中身」の妥当性を保証しない。

### より正確な見解
このサービスは「人生の未来を当てる」商品ではない。成立する商品定義は **“現在の混線を解き、次の実験を設計する意思決定支援”** である。これを実現するための必須修正は以下である。

1) **診断の定義を変更せよ**  
- 「当てる診断」を捨て、「現状の構造化」「選択肢の比較」「実験計画」の品質にコミットする。未来分岐は“予測”ではなく“シナリオ”として提示する。

2) **評価と安全をPhase 0として前倒しせよ**  
- 品質基準（最低限の一貫性、禁則、危機ケースの遮断、根拠提示ルール）を仕様化し、回帰テストの入力セットを作る。これがないLLM運用は必ず崩壊する。

3) **5軸を捨てずに“違和感”を上に載せよ**  
- 5軸の意味を壊す全面改稿は誤りである。違和感は入口のナラティブとして収集し、5軸は骨格として維持する。質問変更は段階的に行い、軸スコアの解釈を守る。

4) **チャネルは一つに絞れ**  
- 初期はココナラ手動に固定し、SOP（収集→確認質問→生成→人間レビュー→納品→フォロー）を確立する。自動化は品質が収束してからである。

この修正を入れた上で初めて、「既存資産（Backtrace/Feedback/DB/検索）」が武器になる。現行計画のまま進める行為は、実装ではなく信用の毀損を作るだけである。
