# Codex (OpenAI GPT-5.2) の批評

### 結論（最初に明言）
最適解は、Claude推奨のOption C（「確率<0.5だけLLM」「0% trigramを埋めるまで補完」）ではない。最適解は **LLMで戦略的にゴールドを増強し（希少trigramを設計的に含める）、確率校正したClassifierを再学習して全件に適用する“アクティブラーニング型ハイブリッド”** である。現状のClassifier出力のままPhase 3同型性を主張するのは誤りである。

### 批判的分析
1. **論理的整合性**
   - 「純卦率 98.2%→1.1%」を成功指標に置くのは誤りである。なぜなら純卦率はラベリングの正しさを保証しないからである。純卦率はノイズ注入でも任意に下げられる。目的は“分布の見栄え”ではなく“正しい遷移の復元”である。
   - 「ユニーク卦34/64」でも「出次数>0が17/64」の時点で、遷移グラフが崩壊している。崩壊したグラフで同型性テストを回して得たp値は、仮説の検証ではなく“前処理の性質の測定”になる。
   - Test Cだけがpro(large)で他がneutral/anti、さらにTest Dは有効ペア2/32で破綻している。これを“同型性の証拠”と総括するのは論理破綻である。最低限「スペクトル指標に限って一致」という限定命題に縮退する。

2. **実現可能性（技術・実務）**
   - TF‑IDF + LogisticRegressionを、200件のゴールドで64卦（実質は各フィールド8クラス×4）へ安定に一般化させる設計は成立しない。なぜなら希少クラスは学習信号が不足し、線形モデルは観測頻度ゼロのクラスを生成できないからである。0% trigramはその帰結である。
   - Claude案の「確率<0.5をLLM」は設計として成立しない。なぜなら **未校正確率** の0.5は誤り率の境界ではないからである。高確信の誤りを大量に残したまま、低確信だけを直しても分布の偏りは残る。
   - 「0% trigramがなくなるまでLLMで補完」は科学的に不正である。なぜなら真の分布が0であるケースまで強制的に非ゼロ化し、以後のテストを“目標に合わせて改竄”するからである。

3. **新規性の真偽**
   - Classifier＋LLMのハイブリッド自体は新規性がない。既存のアクティブラーニング／human‑in‑the‑loopの定石である。新規性を主張すると査読で落ちる。

4. **スケーラビリティ**
   - 全件LLM（Option B）はコストが線形に増える。22,000呼び出しという運用は、品質管理（再現性、再試行、プロンプト固定、監査ログ）まで含めると更に重い。
   - ただしClaude案の“場当たりハイブリッド”はさらに悪い。異なる注釈器（Classifier/LLM）が混在し、誤差構造が不均一になり、検定結果が注釈器の混合比に依存する。これはスケールするほど再現性が壊れる。

5. **反例・エッジケース**
   - **確信度で切るハイブリッドは希少クラスを救えない。** 希少クラスはモデルが“知らない”ため確信度が高い多数派へ吸着する。結果としてLLMに回らず、0%が温存される。
   - **分布0の強制解消は偽陽性を必ず作る。** 乾=0%が真である領域を1件でも人工的に作ると、遷移エッジとスペクトルが汚染される。

6. **代替案との比較**
   - Option A単独（500件）も不十分である。なぜなら「500をランダムに増やす」だけでは希少trigramが増えず、偏りが残るからである。
   - Option D（Phase4へ集中）は合理的な撤退戦略である。ただし撤退の前に「Phase3のテスト設計が注釈誤差に頑健か」を評価せずに捨てるのは、発見（Test C）を自己破壊する。

### Claudeの意見への反論/修正
- 正しい指摘は「34/64カバレッジでは同型性テストに不十分」「LLMは多様性を出す」である。
- 誤りは提案の中心である。
  - **確率<0.5閾値は無意味である。** 校正していない確率は意思決定に使えない。
  - **“0% trigramを埋める”はデータ改竄である。** 目的に合わせて分布を作る行為であり、以後の統計検定を無効化する。
  - **混合注釈の誤差モデルを無視している。** Classifier由来の系統誤差とLLM由来の誤差が混ざると、Test Cの頑健性評価が不能になる。
- 修正案として、LLMは「補修」ではなく「訓練データ設計（希少trigramを含む）」に使うべきである。

### より正確な見解
Phase3を続けるなら、やるべきことは“補完”ではなく“同定”である。具体策は以下である（これ以外は最適解にならない）。

1. **目的関数を修正**：純卦率やユニーク卦数を主指標から外し、各フィールド8trigramの再現率・混同行列・校正誤差（ECE等）を主指標に据える。  
2. **ゴールドを設計して増強（Option Aを正しく実行）**：500件は下限である。ランダム増加は禁止し、4フィールドそれぞれで希少trigramが必ず入るよう層化抽出する。  
3. **モデルを“state_label→trigram”依存から切り離す**：before_lower / before_upper / after_lower / after_upperを別タスクとして直接分類し、64卦は後段で合成する。これで「対応が強すぎる」問題を構造的に潰す。  
4. **確率校正してからアクティブラーニング**：校正済み確率で不確実サンプルを各クラスから均等に選び、LLMで追加ラベル→再学習を反復する。閾値0.5の固定運用は禁止である。  
5. **混合注釈の誤差を推定し、テストに伝播**：LLM再注釈の抜き取り監査を行い、ラベル不確実性をモンテカルロでグラフへ伝播してTest A–Eを再評価する。  
6. **Phase3の主張を縮退**：上記を満たしてもA/B/Dが立たないなら、「同型」ではなく「スペクトル指標での類似」に限定して報告する。限定しない主張は誤りである。

この手順を実行せずにOption Cで“分布を整える”運用に入るなら、Phase3は検証ではなく作話になる。実務としても研究としても失格である。
