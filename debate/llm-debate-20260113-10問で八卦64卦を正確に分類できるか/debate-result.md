# LLM Debate: 10問で八卦64卦を正確に分類できるか

## 議題
10問で八卦64卦を正確に分類できるか

## Claude (Anthropic) の見解
10問では不十分だが工夫次第で実用レベルに到達可能。推奨設計：(1)状況把握3問（現在の立場・課題の性質・緊急度）(2)変化の方向3問（望む変化・避けたい変化・制約条件）(3)内的状態2問（感情・直感）(4)外的環境2問（周囲の反応・資源状況）。各質問は5段階評価ではなく「AかBか」の二項対立で、八卦の陰陽原理と直結させる。ただし限界：複合状況、曖昧な境界ケース、ユーザーの自己認識バイアスは解決困難。

## Codex (OpenAI GPT-5.2) の批評
### 結論（最初に明言）
**10問で八卦64卦を「正確に」分類することは不可能である。**  
10問で実現できるのは「検索・対話の起点として妥当な候補を提示する」レベルであり、単一の正解卦へ高信頼に確定する分類器にはならない。

### 批判的分析
1. **論理的整合性**  
   - 「正確な分類」を主張するには「各ユーザー状態に対応する“唯一の正解卦”が存在する」ことが前提になる。しかし易の卦は**状況記述のラベルではなく、解釈枠組み**であり、同一状況に複数卦が同時に当てはまる。よって前提が誤りである。  
   - 10問→八卦→64卦という決定規則は「生活状況→卦」の写像を固定するが、これは**本来解釈で埋める部分をアルゴリズムで押し切る**設計である。意味論が破綻する。  
   - 64卦は6本の爻で表すが、実務で扱うのは変爻（動く爻）を含む動態である。変爻を扱わない時点で「正確な分類」という語が成立しない。

2. **実現可能性（実装 vs 検証）**  
   - 実装は容易である。問題は**妥当性検証が成立しない**点にある。分類の正確性を測るには「正解ラベル（真の卦）」が必要だが、易の卦に客観的正解は存在しない。よって「正確に分類できる」を検証できず、主張は空語になる。  
   - 12,000件DBがあっても、各事例に対して「その卦が正しい」と言える基準がない限り、教師データにならない。専門家合議でラベリングしても**一致率が低下**し、モデルの上限精度がその一致率で天井打ちになる。

3. **新規性の真偽**  
   - 短い質問票でタイプ分類→コンテンツ提示は既存手法である。新規性は「易の語彙で包んだ」点と「事例DB＋LLM分析」を組み合わせた点にある。  
   - したがって、競争優位は「10問で64卦を当てる」ではなく「事例検索と分析体験の品質」に置くべきである。

4. **スケーラビリティ**  
   - ユーザーが増えるほど入力の多様性（職種・文化・危機度・精神状態）が増え、固定10問は**分布のズレに耐えない**。質問文の解釈差がそのまま分類ノイズになる。  
   - 64クラスに12,000件は平均187件/卦だが、分布は偏る。レア卦は事例が薄く、類似検索が破綻する。規模が増えるほど「当たらない卦」が顕在化する。

5. **反例・エッジケース**  
   - 複合課題（転職＋介護＋離婚など）は単一卦に収束しない。  
   - 本人の自己認識が歪む状況（燃え尽き、抑うつ、躁的高揚、強い不安）では回答が事実ではなく気分を反映し、分類は再現性を失う。  
   - 質問に同意できないユーザー（選択肢が自分を表していない）はランダム回答になり、分類は無意味になる。これは周辺事例ではなく中心的に発生する。

6. **代替案との比較**  
   - 単一卦の確定を捨て、「上位候補（Top-k）＋根拠＋確信度」で提示し、ユーザー選択で収束させる設計は、10問でも成立する。  
   - 「分類」ではなく「検索ランキング問題」として、事例DBの関連度最大化で設計すると、妥当性を行動指標（滞在、納得度、再訪、購入、自己報告の有用性）で評価できる。

### Claudeの意見への反論/修正
- 「二項対立で陰陽原理と直結」は誤りである。陰陽は単なるA/Bの態度選好ではなく、状況の相互作用・力学・時系列の表現である。A/B質問を陰陽に対応させても**対応の妥当性は自動的に生まれない**。  
- 5段階を捨てて二択に寄せる提案は甘い。二択は情報量を増やすのではなく**測定誤差を増やす**。微妙な中間を切り捨て、回答がブレた瞬間に別卦へ飛ぶ。64分類ではこの不安定性が致命傷になる。  
- 4ブロック（状況/変化/内/外）は一般的なコーチング質問の整理であり、易の八卦・爻の意味論へ直接射影できない。ここを「陰陽に直結」と言い切るのは論理の飛躍である。  
- 正しい点は「10問は不十分」と「複合状況・自己認識バイアスは障害」という認識だけである。ただしそれらは“限界”ではなく**本質的な破綻要因**であり、工夫で消えない。

### より正確な見解
- 「10問で64卦を正確分類」は捨てるべき仕様である。理由は単純で、**正解が定義できず、検証できず、再現性が立たない**からである。  
- 目標を「ユーザーにとって有用な事例と洞察へ最短で到達させる」に置き換えるべきである。その上での実務解は次の通り。  
  - 出力を単一卦ではなく「候補上位3〜5卦＋確信度＋質問に対する根拠」にする。  
  - 10問は初期フィルタに限定し、確信度が低いときだけ追加質問を出す（適応型）。  
  - 事例DBは卦で固定検索せず、テキスト特徴（状況・制約・価値観・時間軸）で類似検索し、結果から卦を“説明ラベル”として付与する（分類を目的にしない）。  
  - 検証は「卦の正解率」ではなく、類似事例の関連度、ユーザー納得度、再診断の安定性、行動変化の自己報告で行う。これなら測れる。  

この議題の核心は「10問で足りるか」ではない。**64卦を“正解ラベル”として扱う設計自体が誤りである**。用途を検索・洞察生成に再定義したときだけ、10問は価値を持つ。


## 統合分析

### 補足：実際の分類先は384爻
ユーザーからの指摘により、分類先は64卦ではなく**384爻（64卦×6爻）**であることが明確化された。これにより問題の難易度はさらに上昇する。

| 観点 | 64卦 | 384爻 |
|------|------|-------|
| 分類先数 | 64クラス | 384クラス |
| 10問の情報量 | 2^10 = 1024パターン | 同じ1024パターン |
| 必要な弁別力 | 1024 ÷ 64 = 16倍余裕 | 1024 ÷ 384 = **2.7倍のみ** |
| DB平均事例数 | 187件/卦 | **31件/爻** |

**結論**: 384爻への10問分類は数学的にも破綻する。

2. **複合状況・自己認識バイアス**が障害になることも一致
3. **無料体験で価値を示す**必要性も共通

### 結論・推奨

**Codexの批評を受け入れ、設計を根本から転換する**

#### 旧設計（破棄）
```
10問 → 単一の卦/爻を「正解」として出力 ❌
```

#### 新設計（採用）
```
10問 → 候補Top-5卦 + 確信度 → ユーザー選択 → 爻レベル深掘り → 事例検索 ✅
```

#### 具体的な質問フロー設計

**Phase 1: 初期絞り込み（5問・固定）**
| # | 質問軸 | 目的 |
|---|--------|------|
| 1 | 変化の性質 | 拡大/収縮/維持/転換の4象限で絞る |
| 2 | 主体性 | 自ら動く/受け止める/待つの3態度 |
| 3 | 時間軸 | 急ぎ/中期/長期の3区分 |
| 4 | 関係性 | 個人/組織/対外の3スコープ |
| 5 | 感情基調 | 積極/慎重/不安/楽観の4象限 |

**Phase 2: 適応型質問（0〜5問・確信度に応じて）**
- Phase 1で確信度 < 70%の場合のみ追加質問
- 候補卦の弁別に特化した質問を動的生成

**Phase 3: ユーザー選択**
- Top-5候補を「状況説明」として提示（卦名は見せない）
- 「これが近い」を選ばせる → 爻レベルへ深掘り
- 爻は「状況の段階」として説明（1爻=初期、6爻=終末期など）

**Phase 4: 事例提示**
- 選択された卦×爻に紐づく類似事例を検索
- **無料版**: 事例件数と概要のみ
- **有料版**: 詳細事例3件 + Claude分析

#### 検証指標（正解率ではなく）

| 指標 | 測定方法 | 目標値 |
|------|----------|--------|
| 関連度 | 類似事例への「これは参考になる」率 | 60%以上 |
| 納得度 | 診断結果への5段階評価 | 平均3.5以上 |
| 再現性 | 同一ユーザーの再診断での候補一致率 | 70%以上 |
