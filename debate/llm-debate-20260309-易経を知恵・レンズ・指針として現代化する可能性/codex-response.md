# Codex (OpenAI GPT-5.2) の批評

### 結論（最初に明言）
易経を「知恵・レンズ・指針」として提供するサービスは成立する。だが、**占い・予測・成功率の物語を少しでも混ぜた時点で破綻する**。価値は「当てる」ではなく、**状況を型に圧縮し、行動の禁忌と優先順位を提示する“解釈のエンジン”**にある。

### 批判的分析
- **論理的整合性（最大の穴）**  
  「予測は否定された」「因果的予測は不可能」という確定事項がある以上、DBや確率テーブルを使って「成功率が高い」「この行動が効く」を言った瞬間に自己矛盾である。ここを曖昧にすると、プロダクトの信用はゼロになる。
- **“知恵”の定義が未確定で、要件が溶けている**  
  「知恵・レンズ」と言うだけでは、生成物が汎用アドバイスと区別できない。知恵は少なくとも次の4種に分解して仕様化しなければならない：  
  1) 状態の同定（いま何が起きているか）  
  2) 行動制約（いま“やるな”）  
  3) 優先順位（いま“これをやれ”）  
  4) 視点変換（同一状況の別解釈）  
  これを定義しない構想は、AIコーチングの薄味コピーで終わる。
- **実現可能性：入力→卦/爻の割当が最大コストであり、単一断定は誤り**  
  kappa=0.561が天井という事実は、「状況ラベリングは本質的に揺れる」ことを示す。ゆえに「あなたは水雷屯だ」と単一ラベルで断定する設計は誤りである。正しい設計は、**複数候補レンズ（例：上位3つ）＋割当根拠＋ユーザーの軽量な選択**である。ここを避けると、外れた瞬間に“占い以下”になる。
- **DB（11,336件）の使い方を間違えると毒になる**  
  11,336件は「事例集」としては資産だが、「一般法則」や「成功率」の根拠にはならない。観測データは交絡と選択バイアスが支配し、さらにアノテーション誤差が乗る。従ってDBの正しい役割は、**因果の証明ではなく、典型パターン・失敗様式・転機の兆候のカタログ化**である。
- **“質問しない”要件を文字通りに取ると品質が崩壊する**  
  自由記述1発で深い指針を出すのは不可能である。質問を嫌うのは「内省を迫る問い」への拒否であり、**事実パラメータの取得（領域、制約、時間軸、利害、可逆性）**は不可欠である。解は「問いかけ」ではなく**フォーム化**である。
- **新規性の真偽：易経“風”は新しくない。防衛線は“説明責任”だけである**  
  AI×占術は既に飽和している。新規性を主張するなら、(1) 64×6の状態空間で一貫した分類、(2) 互卦/錯卦/綜卦で体系的な視点変換、(3) DB事例と古典テキストへのトレーサビリティ、の3点をプロダクトで実装して初めて成立する。
- **スケーラビリティ：技術より運用がボトルネック**  
  スケール時に壊れるのは推論基盤ではなく、解釈品質の一貫性、危険領域（医療・金融・自傷）の安全設計、クレーム対応、説明の再現性である。ここに人件費が乗る。
- **反例・エッジケース（破綻ポイント）**  
  1) ユーザーが「予言」を求める／依存する  
  2) 高リスク意思決定（投資・離婚・医療）に使う  
  3) 出力がバーナム効果（誰にでも当てはまる文）になる  
  4) 多レンズ提示が“情報過多”になり行動が止まる  
  これらを設計で潰さない限り、プロダクトは社会的に有害になる。
- **代替案との比較：易経である必然を証明しないと負ける**  
  変化の指針なら現代フレーム（状況判断、意思決定、システム思考）で代替できる。易経で勝つ必然は「象徴体系×段階性×反転視点」が**一つの統語論**になっている点だけである。ここをプロダクト価値に変換できないなら、易経を選ぶ理由がない。

### Claudeの意見への反論/修正
- **「同じ状況で耐える・潜伏を選んだ事例は成功率が高い」**は誤りである。なぜなら、因果的予測が不可能だと確定している以上、成功率を“行動の推奨根拠”として語れないからだ。言えるのは「記述的にそういう事例が多い」までであり、しかもバイアスとラベル誤差付きである。
- **「384爻はタロットより精緻」**は評価軸が誤っている。分類数の多さは価値の証明にならない。細分化はユーザーの迷いと恣意的当てはめを増やし、むしろ品質を落とす。
- **「ChatGPTにはできない」**は誤りである。汎用LLMは易経“風”の助言を生成できる。差別化は生成能力ではなく、**一貫した分類体系・根拠の提示・事例参照・再現性**である。
- **「占い市場の知的層」狙い**は甘い。占いの主要価値は「断定」と「情緒」であり、あなたが捨てた領域である。狙うべきは占い客ではなく、**意思決定の言語化・状況把握の型**に金を払う層である（個人でも組織でも）。
- 一方で、**「問いではなくレンズ」への修正**と、互卦/錯卦/綜卦を“視点変換装置”として使う発想は正しい。ここは筋が通っている。

### より正確な見解
- **成立形（UI/UX）：1画面で“圧縮された解釈レポート”を返す**  
  入力は「内省質問」ではなく**状況パラメータ**に限定する（例：領域、時間軸、制約、可逆性、関係者数、緊急度）。  
  出力は固定フォーマットでよい：  
  1) 状況の型（主レンズ：卦＋段階）  
  2) いまの禁忌（やるな）  
  3) いまの推奨（やれ）  
  4) 転機の兆候（次に何が起きたら段階が変わるか）  
  5) 別角度レンズ（互/錯/綜を最大2つに絞る）  
  6) 類似事例（DBから3件、結末ではなく“分岐点”を提示）  
  7) 古典根拠（短い要約＋引用は最小）  
  これで「問いかけなし」に近い体験を維持しつつ、情報としての密度が出る。
- **差別化は“体系と根拠の提示”でのみ成立する**  
  生成文だけなら汎用AIに負ける。勝ち筋は、(a) 64×6の状態空間での一貫した運用、(b) 視点変換の規則、(c) DB事例と古典テキストへのトレーサブルな紐付け、(d) 単一断定を避けた複数候補提示、である。
- **DB×古典で生成できる“知恵”は因果ではなく行動設計である**  
  生成物は「成功する方法」ではなく、  
  - よく起きる誤謬（この状態でやりがちな失敗）  
  - 先に整える順序（段取り）  
  - リスクの局所化（被害を限定する手順）  
  - 反転視点（同じ出来事の別読みによる行動選択肢の拡張）  
  これらに限定すべきである。ここを守ると、予測否定と矛盾しない。
- **限界・リスクは“権威化”である。対策は“非断定化と説明責任”である**  
  易経を権威として語ると依存と誤用が必ず発生する。対策は、出力を「断言」ではなく**条件と制約として記述し、割当根拠と代替レンズを併記する**ことだ（ユーザーに“選べる余地”を残す）。
- **マネタイズはできるが、売り方を間違えると死ぬ**  
  予言・的中で売るのは破綻する。売れるのは「意思決定支援の型」としてである。具体策は、  
  - **B2B**：チームの状況診断・プロジェクト局面のレンズ（定例に組み込む）  
  - **B2C**：日誌・振り返り・行動チェックリスト（習慣導線が必須。これがないサブスクは成立しない）  
  - **API/ライセンス**：64×6の状態モデルと事例索引の提供  
  この路線なら、占い市場の論理に巻き込まれずに収益化できる。
