# LLM Debate: 易経変化ロジックDBにおける企業事例と個人事例の混在問題：分離すべきか、統合のまま改善すべきか

## 議題
易経変化ロジックDBにおける企業事例と個人事例の混在問題：分離すべきか、統合のまま改善すべきか

## Claude (Anthropic) の見解
物理的分離ではなく論理的分離を推奨。(1)scaleフィルタリングをBacktraceEngineのデフォルトに、(2)個人事例にexpertise_levelフィールド追加、(3)事例検索時にscale+expertise_levelでマッチング、(4)カテゴリ跨ぎ事例は参考情報として別枠表示、(5)行動パターン確率計算はカテゴリ別に分離。ただし完全分離はカテゴリ内事例数減少と構造的類似性喪失のリスクあり。現状のデータ品質格差(企業S/A:69% vs 家族:27%)と行動パターン分布差(個人の攻める39% vs 家族11%)は、混在させたまま分析すると誤った推奨を生む。実例：タクシー運転手の日常売上回復にJAL破綻を参考事例として提示してしまった。

## Codex (OpenAI GPT-5.2) の批評
### 結論（最初に明言）
企業事例と個人事例を同一母集団として統合し、同じ検索・同じ確率テーブルで推奨に投入する設計は誤りである。推奨生成・確率推定・評価は`scale`（company/individual/family/country/other）条件付きに分離し、運用事故を防ぐためストレージも別テーブル/別インデックスに物理分離するべきである。カテゴリ跨ぎは推奨根拠から除外し、抽象化したパターン解説としてのみ扱う。

### 批判的分析
- **論理的整合性**  
  混在DBは「同じ状態・同じ卦・同じ行動タイプなら同じ意味を持つ」という暗黙前提に依存している。この前提は誤りである。なぜなら行動パターン分布がカテゴリ間で大きく異なる（例：個人の「攻める・挑戦」39.3% vs 家族11%、家族の「守る・維持」26.7% vs 企業13.8%）ため、`P(action | state, hex)`は`scale`に強く依存する。`scale`を無視した混合確率は統計的に無意味で、誤推奨を必ず生む。タクシー運転手にJAL破綻を提示した事例は、その破綻が既に顕在化している証拠である。  
  さらに「対話・融合」など同一ラベルがスケールで別概念に化けている。これは検索の類似度も確率推定も同時に汚染するラベル多義性であり、フィルタリングだけでは根治しない。

- **実現可能性**  
  実装できるのは分離であり、実装破綻するのは曖昧ラベル（`expertise_level`等）を追加して運用で吸収する案である。`scale`条件付きにすると逆引きインデックス（`rev_*`）と確率テーブル（`before_state×before_hex`等）に`scale`次元を入れる必要があるが、これは再計算・再構築の工数で済む。最大コストは「ラベル定義と監査」であり、`expertise_level`はそこを直撃する。  
  また品質格差（企業S/A 68.9% vs 家族27.4%、国家verified 72.8% vs 家族21.5%）を重みに入れずに統合推定する設計は誤りである。品質は検索・推定の両方で重みとして扱う必要がある。

- **新規性**  
  「scaleで分ける」「カテゴリ別に確率を出す」は新規性ではない。推薦・意思決定支援の基本衛生である。新規性を語るなら、階層化（部分プーリング）・不確実性提示・行動オントロジー再定義まで含めるべきである。

- **スケーラビリティ**  
  統合のまま件数を増やすと支配カテゴリ（企業5,510件）が検索・統計を占有し、個人ユーザー向け最適化が劣化する。これは「データが増えれば改善する」という期待に反して、誤差が増幅する構造である。

- **反例・エッジケース**  
  自営業・家族経営・小規模事業は「個人/家族/企業」の境界が曖昧で、単純な離散`scale`だけでは分類が破綻する。したがって`scale`分離に加えて「当事者数」「資源制約」「時間軸」「可逆性」「合意形成コスト」など、意味差を生む特徴量が必要である。

- **代替案との比較**  
  二択（完全統合 vs 完全分離）が誤ったフレーミングである。正解は「推奨は分離、学習は階層統合」である。推奨時は`scale`をハード制約にし、統計推定は共通priorで部分プーリングして情報を共有する。

### Claudeの意見への反論/修正
- **(1) scaleフィルタをデフォルトにする**  
  「デフォルト」は安全性担保にならない。1箇所のフィルタ漏れで事故が再発する。`scale`は推奨APIの必須引数として強制し、未指定なら推奨を返さない設計が正しい。

- **(2) 個人事例にexpertise_levelを追加**  
  誤りである。なぜならexpertiseはユーザー属性であり、事例属性ではない。事例に付けるなら「その行動を遂行するのに必要な能力（required_capability）」であり、定義と測定が別物になる。4段階ラベルは恣意的で、ラベルノイズを増やして精度を落とす。

- **(3) scale+expertiseでマッチング**  
  目的（スケール不一致の誤推薦防止）に対して手段がずれている。誤推薦は`scale`のハード制約で止まる。必要なのは曖昧ラベル追加ではなく、行動の実行条件（当事者数・資源・時間・可逆性）の表現である。

- **(4) 跨ぎ事例を参考情報として別枠表示**  
  別枠でもユーザー認知を汚染する。参考枠が有用になるのは、事例物語ではなく「共通構造だけを抽象化して提示する」場合に限る。跨ぎはデフォルト非表示、要求時のみ抽象パターンとして出すべきである。

- **(5) 確率をカテゴリ別に分離**  
  方向は正しいが不十分である。家族787件のような少数カテゴリを単純分離すると推定が不安定になる。階層化（部分プーリング）と不確実性提示が必須である。

### より正確な見解
- **本質**  
  問題は「物理的に一緒に置いている」ことではなく、「母集団の混合」と「行動ラベルの多義性」を推奨ロジックに直結させていることである。

- **正しい方針**  
  1) 推奨・逆算（L1–L3）・確率テーブル・インデックスを`scale`条件付きにする。  
  2) 物理分離（別テーブル/別インデックス）で混在事故を構造的に不可能にする。  
  3) 統計推定は階層化で共通部分だけ共有し、少数カテゴリ（家族）を維持しつつ過学習を止める。  
  4) 信頼度S/Aやverifiedを重みとして組み込み、品質格差を推奨に反映する。  
  5) 跨ぎは「事例」ではなく「抽象パターン」を提示し、推奨根拠には混ぜない。

- **家族787件**  
  削除は誤りである。分布が異なる以上、統合は誤推奨を増やす。階層化＋品質重み＋不確実性提示で「使えるが過信しない」形に落とすべきである。

- **情報量の評価**  
  単純分離で失うのは、混合汚染による擬似相関である。共有すべき構造は階層化で保持できる。混合のまま得る統計は構造的に誤差を含み、情報ではない。


## Gemini (Google) の見解

### 結論
**論理的分離を強く推奨。物理的分離は避けるべき。**
易経の持つ「マクロとミクロの相似性（フラクタル構造）」という強みが失われるため。

### 技術調査結果
- `CaseSearchEngine` のインデックスに `scale` が含まれていない（混在の直接原因）
- `rev_after_state.json` 等の逆引きインデックスにカテゴリ情報が完全に欠落
- `build_reverse_indices.py` が全カテゴリをフラットに合算して集計

### 提案: 階層型論理分離（Hierarchical Logical Separation）
1. インデックス構造を `{ "scale": { "key": { statistics } } }` に刷新（`all` キーで後方互換性維持）
2. 階層的フォールバック: 同一scale+expertise → 同一scale全体 → all(グローバル)
3. 少数カテゴリはベイズ平滑化（カテゴリ統計50% + グローバル統計50%）
4. カテゴリ跨ぎは「構造的類似事例」バッジ付きで別枠表示

### 代替アプローチ: Analogy Scoring
`TotalScore = BaseChangeSimilarity * ScaleMatchWeight * ExpertiseMatchWeight`
ハードフィルタではなくソフトな重み付けで自然にスケール不一致を抑制。

### expertise_levelについて
4段階（novice/intermediate/advanced/expert）は適切と判断。個人の意思決定の質的差異を捉える。

### 相違点（重要な論争）

#### 論争1: 物理分離 vs 論理分離
- **Codex**: 物理分離を主張。「1箇所のフィルタ漏れで事故が再発する」。構造的に混在事故を不可能にすべき
- **Gemini**: 論理分離を主張。「物理分離はフラクタル構造の強みを失い、重み付き補完が困難になる」
- **判定**: Codexの「安全性の構造的担保」は正しい指摘。ただし物理分離のメンテナンスコスト増も事実。**妥協案: scale別の逆引きインデックスファイル（物理分離）+ 統合JSONLファイル（データソースは統合維持）**

#### 論争2: expertise_level の是非
- **Codex**: 反対。「expertiseはユーザー属性であり事例属性ではない。required_capabilityなら分かる」
- **Gemini**: 賛成。「初心者の売上回復と熟練者の市場開拓は質的に異なる」
- **判定**: Codexの指摘は本質的。事例に付けるなら「その行動の実行に必要な条件」であるべき。ただし既存データ3,217件へのラベル付けコストを考慮すると、当事者数・資源制約等の客観的特徴量を優先し、expertise_levelは後回しが妥当

#### 論争3: カテゴリ跨ぎの扱い
- **Codex**: デフォルト非表示。「別枠でもユーザー認知を汚染する」
- **Gemini**: バッジ付きで表示。「相似形の示唆として価値がある」
- **判定**: 初期実装ではCodexの「デフォルト非表示」を採用し、ユーザーが明示的に要求した場合のみ抽象パターンとして表示。UXテストで有用性を検証してから表示範囲を拡大

#### 論争4: ハードフィルタ vs ソフトスコアリング
- **Codex**: scaleは必須引数（ハード制約）
- **Gemini**: ScaleMatchWeight（ソフトスコアリング）
- **判定**: 両立可能。推奨APIはscale必須（Codex案）としつつ、scale内の類似度計算にはソフトスコアリング（Gemini案）を適用

### Codexからの重要な新規指摘（Claude案に欠けていたもの）
1. **ラベル多義性問題**: 「対話・融合」がスケールによって全く異なる意味を持つ。フィルタリングだけでは根治しない。行動オントロジーの再定義が必要
2. **部分プーリング**: 完全分離と完全統合の二択ではなく、「推奨は分離、学習は階層統合」が正解
3. **エッジケース**: 自営業・家族経営は既存5カテゴリの境界が曖昧。追加特徴量が必要
4. **品質重み**: 信頼度S/A・verified率を推奨の重みに反映すべき（現在は無視されている）

### Geminiからの重要な新規指摘（Claude案に欠けていたもの）
1. **段階的フォールバック**: scale+expertise → scale全体 → グローバルの3段階検索
2. **ベイズ平滑化**: 少数カテゴリにはカテゴリ統計+グローバル統計の重み付き平均
3. **Analogy Scoring**: ハードフィルタの代わりにソフトな重み付けスコアリング
4. **インデックス構造の具体案**: `{ "scale": { "key": { statistics } } }` + `all` キー
