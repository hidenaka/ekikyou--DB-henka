# Codex (OpenAI GPT-5.4) の批評

### 結論（最初に明言）
**二者択一なら、現時点で採るべきは既存の8質問パイプラインである。6軸への全面再設計は不採択である。Claudeのハイブリッド案も不採択である。**

理由は単純だ。8質問はすでに動く資産であり、6軸はまだ測定体系として存在していない。しかも両者は同じ層の質問ではなく、**別の潜在空間を持つ競合アーキテクチャ**である。混ぜれば良くなる設計ではない。

### 批判的分析
- **論理的整合性**
  - 既存8質問は「卦を先に決め、そこから勢・時・状態・行動推奨へ流す」設計である。
  - 6軸案は「意思決定条件を先に取り、卦は後から物語として載せる」設計である。
  - この2つは補完関係ではない。**分類の起点が逆**である。両方を同時採用すると「現在地」が二重定義になる。設計として破綻である。
  - `336テスト通過` はコードの安定性を示すだけで、質問設計の妥当性は示さない。ここを混同してはいけない。

- **実現可能性**
  - 6軸全面再設計には、11,336件の事例を6軸で再ラベル化する作業が必要である。
  - 既存の `before_state / trigger_type / action_type` は6軸の代替ではない。`時間圧 / 可逆性 / 外部制約 / 資源余力 / 対人摩擦 / 主体性` を直接表していない。
  - したがって、6軸は質問を足せば済む話ではない。**検索索引、類似度計算、説明生成、品質評価を全部作り直す話**である。
  - kappa=0.561の状態で軸を増やすのは危険である。粒度を上げるほど誤差が増幅する。

- **新規性の真偽**
  - 6軸問診は新規性の源泉ではない。意思決定支援としては筋が良いだけで、方法論としては標準的である。
  - 逆に8質問も新規ではない。新規性は質問票ではなく、**11,336件の事例DBと易経の多視点解釈を接続している点**にある。

- **スケーラビリティ**
  - 8質問単独はスケールする。列挙型入力、既存テーブル、既存テストで回る。
  - 6軸単独は将来的には強いが、初期整備コストが重すぎる。
  - ハイブリッドは最悪である。質問数、矛盾処理、説明責任、品質管理が全部増える。**最も非効率**である。

- **反例・エッジケース**
  - 漠然と不安なユーザーは、可逆性や外部制約を最初から正確に答えられない。6軸強制入力は入口で詰む。
  - 一方、8質問は混合状態に弱い。単一の `primary_hex` に圧縮すると、多層問題を落とす。
  - したがって、正解は全員に固定質問票を当てることではない。**自由記述を先に取り、曖昧な軸だけ後から聞く適応型**である。

- **代替案との比較**
  - 本当に優れた案は「8質問をそのまま出す」でも「6軸へ全面移行」でもない。
  - 既存8質問を下流エンジンとして温存し、入口だけをナラティブ化し、6軸はシャドー評価する。この二段階移行が最も合理的である。

### Claudeの意見への反論/修正
- **「ハイブリッドが最も効率的」は誤りである。**  
  効率的ではない。最も統合コストが見えにくいだけである。
- **「全面再設計は不要」は正しい。**  
  既存資産を捨てるのは愚策である。
- **「6軸を追加質問として組み込む」は誤りである。**  
  追加した瞬間に、どちらを主判定にするのか、矛盾時にどう裁定するのか、既存統計テーブルへどう接続するのか、3つの未解決問題が発生する。
- **「非象徴タグが一部あるから足せる」は甘い。**  
  それらは6軸の代理変数ではない。意味空間が違う。
- **「根本ロジックは完成している」という評価も甘い。**  
  完成しているのはコードであって、現サービス方針との整合ではない。現方針は「予測しない」「象徴解釈と事例参照を分離」「成功率表示禁止」であり、既存8質問はまだその契約に最適化されていない。

### より正確な見解
**今やるべきことは、8質問を本番基盤として残し、6軸を本番に混ぜず検証専用に隔離することである。**

- 入口はまず「今、何を決めたいか」の自由記述に変える。
- 8質問は全問固定ではなく、曖昧な項目だけ確認する補助層に下げる。
- 出力は既存の内部スコアをそのまま見せず、合意済みの5点固定に再編する。
- 6軸を採用したいなら、本番導入前に `自己認識一致率`, `類似事例の納得度`, `人手レビュー修正率`, `行動提案採用率`, `追加質問発生率` で既存系と比較しろ。

**結論をもう一度言う。今は6軸へ作り直す局面ではない。8質問基盤で進め、6軸はシャドー評価で勝ったら置換する。それ以外は設計ではなく混乱である。**
