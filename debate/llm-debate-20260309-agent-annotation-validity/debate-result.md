# LLM Debate: agent-annotation-validity

## 議題
agent-annotation-validity

## Claude (Anthropic) の見解
APIクレジット不足のため、Claude Codeサブエージェントが直接800件のアノテーションを実行する代替案を検討中。エージェントはcases.jsonlからテキストを読み、プロンプトテンプレートに従って4つの八卦フィールドを独立に判定する。二重注釈は2つの独立したエージェントで実行する。この方式はスクリプト方式と比べて温度制御やJSON出力の強制ができないデメリットがあるが、(1)同一モデルファミリーによるアノテーション品質は同等、(2)エージェント間のコンテキスト分離により独立性は確保される、(3)コスト面ではClaude Codeサブスクリプション内で追加課金なし、という利点がある。バッチサイズは50件/エージェントで16バッチ×2パス=32エージェント起動が現実的。

## Codex (OpenAI GPT-5.2) の批評
### 結論（最初に明言）
この「チャット（サブ）エージェントによる直接アノテーション」は**ゴールドセット構築として不適格である**。得られるのはゴールドではなく、せいぜい**監査可能な“シルバー（擬似ゴールド）”**である。ゴールドを名乗るなら、**形式・再現性・独立性・監査・最終審査**の要件を満たす必要があり、本案は満たしていない。

---

### 批判的分析

1. **論理的整合性（主張と根拠の矛盾）**
   - 「同一モデルファミリーだから品質同等」は誤りである。品質を決める一次因子は**制約（JSON強制・スキーマ検証・温度・再試行）と運用（resume・ログ・逸脱検知）**であり、モデル名ではない。スクリプト方式はこれを設計で担保しているが、エージェント方式は担保していない。
   - 「コンテキスト分離＝独立性確保」は誤りである。独立性とは**誤りが相関しない**ことであり、別インスタンスで動かしても**同一モデル・同一判断規範・同一バイアス**なら誤りは相関する。コンテキスト分離は「情報リーク防止」には効くが、「独立性」には効かない。
   - 「サブスク内で追加課金なし＝コスト優位」は議論のすり替えである。金銭課金がゼロでも、**実務コスト（手動バッチ運用、逸脱修正、再実行、品質監査、上限・速度制約）**は増大する。

2. **実現可能性（技術・実務の隠れコスト）**
   - **JSON非強制**は致命的な運用リスクである。下流が機械処理（集計・一致率・kappa・学習）なら、**一件でも形式逸脱した時点でパイプラインが壊れる**。スクリプト方式は最初からここを潰している。
   - **--resume相当がない手動運用**は、800件×2パスという量に対して不適格である。途中失敗・重複・欠落・バージョン混在が必ず混入し、後から追跡できないデータになる。
   - 温度制御不可は、再現性・監査性を損なう。再現できない教師データは、評価基準として機能しない。

3. **新規性の真偽**
   - 新規性はない。単に「APIスクリプト」を「対話UI」に置き換えただけで、方法論は同一である。研究上の寄与は発生しない。

4. **スケーラビリティ**
   - 32回のサブエージェント起動（50件×16バッチ×2パス）は、規模拡大に耐えない。1,600件（800×2）を**手動で監査・修正・再実行**する運用は破綻する。
   - バッチ間ドリフトは構造的に起きる。基準が文書化・固定化されず、各エージェントが“その場の解釈”で走る設計だからである。

5. **反例・エッジケース**
   - テキストが矛盾（before/afterの整合不良）・不足（状態が曖昧）・複合（複数変化が混在）するケースで、エージェントは**強制選択して誤ラベルを量産**する。回避するには「不明」「判定不能」等の**逃げラベル**と再審査フローが必須である。
   - 「理由（rationale）」は品質保証にならない。LLMはもっともらしい説明を生成でき、説明が正しいことの証拠にならない。

6. **代替案との比較**
   - 同じ制約下でも、より優れた代替は存在する。具体的には「エージェントを使う」こと自体ではなく、**スクリプト方式が持っていた統制（スキーマ検証・再試行・ログ・resume）をローカル側で復元**することが本筋である。

### Claudeの意見への反論/修正
- (1) **「同一モデルファミリーで品質同等」**は誤りである。品質差はモデル差より**運用設計差**で決まる。本案は運用設計が弱く、同等品質を主張する根拠がない。
- (2) **「コンテキスト分離で独立性は確保」**は誤りである。独立性の核心は**誤り相関の低減**であり、同一モデル二重注釈は相関を下げない。二重注釈の価値（誤り検出・信頼性推定）が大幅に失われる。
- (3) **「追加課金なしでコスト優位」**は甘い。金銭課金がゼロでも、手動運用と品質監査の工数が増え、総コストは上がる。これは研究・開発の現場で必ず問題化する。
- (4) **「50件/エージェントが現実的」**は雑である。現実性の評価軸は「起動回数」ではなく、**逸脱率・再実行単価・監査容易性**である。50件は再実行単価が高すぎ、壊れた時の損失が大きい。

### より正確な見解

1. **このアプローチは妥当か？**
   - **ゴールドセット目的には妥当でない。**  
     ゴールドが「評価基準としての高信頼ラベル」を意味するなら、人間監査と再現性が必須であり、本案は欠けている。
   - **暫定シルバーセットとしては妥当である。**  
     ただし名称・用途を「擬似ゴールド」「銀ラベル」に改め、評価解釈（上限性能の推定、相対比較の補助）に限定する必要がある。

2. **二重注釈の独立性欠如は致命的か？**
   - **致命的である。**  
     独立性がない二重注釈は「一致＝正しい」という推論を支えない。同一モデル二重注釈は、同じ誤りに二人で同意する構造を持つためである。  
     例外は「一致率の低い項目を抽出する」という用途だけであり、この場合は“独立性”ではなく“矛盾検出”として扱う。

3. **品質担保のために追加すべき対策（必須）**
   - **スキーマ固定＋機械検証**：許容値（八卦の8値、欠損時のラベル）を厳密定義し、JSONをローカルで検証して**不合格は再実行**する。
   - **プロンプトの凍結とハッシュ管理**：各パスの指示文・例示・定義を固定し、`prompt_version`を出力に埋め込む。これがないデータは監査不能である。
   - **キャリブレーション（アンカー）**：各バッチ冒頭に少数の基準ケース（手動確定）を入れ、ズレたらそのバッチを破棄する。
   - **第三者（第三パス）による裁定**：不一致ケースは第三者（別モデル、または人間）で裁定する。同一モデルの自己裁定は独立性を回復しないため、最終裁定者を変える。
   - **人間の抜き取り監査**：一致ケースも含めてランダムに監査し、エラー率を推定する。これがない「ゴールド」は成立しない。

4. **バッチサイズとエージェント分割の推奨**
   - **50件/バッチは不適切である。**再実行損失が大きく、注意資源が分散し、逸脱修正が困難になるためである。
   - 推奨は**20〜25件/バッチ**である（監査容易性と起動回数の両立点）。  
   - 分割は「ケースID連番」ではなく、**難易度・長文/短文・欠損率で層化**して各バッチを均質化する。これをしないとバッチ間品質が比較不能になる。

以上。ここまでの条件を満たさないなら、この案は「APIクレジット不足の穴埋め」でしかなく、ゴールドセットの目的（信頼できる評価基盤）を破壊する。用途と呼称を下げ、統制と監査を追加して初めて成立する。
