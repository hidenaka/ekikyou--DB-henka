## 1. 概要

### 1.1 サービス名
**人生の現在地** — 意思決定支援サービス

### 1.2 対象
「このままでいいのか」を感じている個人。外から見ると順調だが、内側に違和感を抱えている人。

### 1.3 設計原則
- 易経は内部フレームワークとしてのみ使用し、レポート上には一切表示しない
- 類似事例DB（11,336件）から事例を検索し、根拠として使用する
- 確率・予測・断定は一切行わない
- ココナラで手動納品（PDF/テキスト形式）

### 1.4 レポートの版
- **フル版**: 6セクション全体、3,000-5,000字
- **軽量版**: セクション1(短縮版) + セクション4(簡易版) + セクション5、1,000-2,000字

---

## 2. 入力データ

レポート生成には以下の入力が必要:

| 入力 | ソース | 必須 |
|------|--------|------|
| ユーザーのナラティブ（原文） | Step 1: ナラティブ入力 | 必須 |
| 構造化抽出結果（JSON） | `extraction_prompt_individual.md` の出力 | 必須 |
| 要約合意テキスト | Step 5: 要約合意ゲートで合意済みの要約 | 必須 |
| 類似事例リスト（JSON） | CaseSearchEngine の出力 | 必須 |
| サービスプラン | `full` または `light` | 必須 |

### セクション1: 現在地の構造化（800-1200字）

**目的**: ユーザーの違和感を整理し、言語化する。分析ではなく整理。

#### テンプレート

```
■ あなたの現在地

[ユーザーのナラティブから構造化した状況の要約。要約合意ゲートで確認済みの内容をベースに拡張する]

○ 状況の整理
- 現在の状態: [current_stateの自然言語化 — ラベル名は使わない]
- エネルギーの方向: [energy_directionの自然言語化]
- 変化のきっかけ: [trigger_natureの自然言語化]
- 今のフェーズ: [phase_stageの自然言語化]

○ 制約と条件
- [constraintsの各項目を自然言語で列挙]
- リスクへの姿勢: [risk_toleranceの自然言語化]

○ 見えている選択肢
- [choice_setのidentified_choicesを列挙]
- [hidden_choicesがあれば「他に考えられる可能性」として追記]
```

#### 記述ルール
- ユーザー自身の言葉をできるだけ引用する（「」で囲む）
- 構造化ラベル名（current_state, energy_direction等）は見せない
- 分析的にならない。整理に留める
- 「あなたは〇〇タイプです」等のラベリング禁止
- 要約合意ゲートで確認済みの内容と矛盾しないこと

#### 自然言語化の例

| 構造化ラベル | 自然言語化の例 |
|-------------|---------------|
| current_state: 安定だが違和感 | 「外から見ると大きな問題はないが、内側にモヤモヤとした違和感がある状態」 |
| energy_direction: 内向・収束 | 「エネルギーが内側に向かい、自分自身と向き合っている段階」 |
| trigger_nature: 漸進的変化 | 「突発的な出来事ではなく、時間をかけて少しずつ違和感が積み重なってきた」 |
| phase_stage: 始まり | 「変化の入り口に立っている段階。まだ動き出してはいないが、意識は動き始めている」 |
| risk_tolerance: レベル2 | 「できれば大きなリスクは避けたいという姿勢」 |

#### 軽量版（light）の場合
- 400-600字に短縮
- 「状況の整理」のみ記載。「制約と条件」「見えている選択肢」は省略

### セクション2: このまま進んだ場合のシナリオ（600-1000字）

**目的**: 現状維持した場合のパターンを類似事例から提示する。

#### テンプレート

```
■ このまま進んだ場合

[current_state + energy_direction + intended_action="現状維持" での類似事例パターンの概要]

○ 類似の状況にいた人たちのその後
[高類似事例(>=0.70)から抽出した「現状維持」選択の結果パターン]

事例A（[根拠ラベル]）: [匿名化した事例概要、50-80字]
事例B（[根拠ラベル]）: [匿名化した事例概要、50-80字]
事例C（[根拠ラベル]）: [匿名化した事例概要、50-80字]

○ 共通して見られる傾向
- [パターン1]
- [パターン2]

※ これらは類似事例から見られた傾向であり、
  あなたの未来を予測するものではありません。
  個別の状況により異なります。
```

#### 記述ルール
- 根拠ラベル = 「観測済み」または「仮説」（`evidence_specification.md` 準拠）
  - **観測済み**: credibility_rank が S/A で、アウトカムが確認された事例
  - **仮説**: credibility_rank が B、またはアウトカム未観測の事例
- 事例は必ず3件以上。3件未満の場合「**事例数不足: 仮説としてのみ提示**」を明記
- 事例は匿名化必須（`evidence_specification.md` セクション5の匿名化ルール準拠）
  - 個人名 → 「Aさん」「ある経営者」等
  - 企業名 → 公知情報のみ可。非公知は「同業界のB社」等
  - 時期 → 「2010年代後半」等に幅を持たせる
  - 金額 → 「数千万円規模」等に丸める
- 「〇〇になります」等の断定禁止
- 確率・パーセンテージ禁止
- 末尾の注意書きは必ず付与する

#### 事例概要の書き方
- 50-80字で簡潔に
- 構造: 「[状況] → [選択] → [結果]」の流れ
- 例: 「メーカー勤務10年目、違和感を抱えつつ現状維持を選択。3年後、部署異動がきっかけで環境が変わった（観測済み）」

### セクション3: 別ルート 2-3本（800-1200字）

**目的**: 異なる選択をした類似事例からのパターン提示。

#### テンプレート

```
■ 別のルート

[choice_setのidentified_choicesまたはintended_actionから2-3本のルートを提示]

── ルート①: [ルート名、5-10字]

[この選択をした類似事例のパターン]

事例D（[根拠ラベル]）: [匿名化した事例概要]
事例E（[根拠ラベル]）: [匿名化した事例概要]

前提条件: [このルートが成立する条件]
注意点: [このルートのリスクや考慮事項]

── ルート②: [ルート名]

[同様の構造]

── ルート③: [ルート名]（あれば）

[同様の構造]

※ ルートの優劣はつけていません。
  どれが「正解」ということはなく、
  それぞれに異なる前提条件とリスクがあります。
```

#### 記述ルール
- ルートの優劣をつけない（「おすすめ」「最適」「ベスト」禁止）
- 各ルートに最低2件の事例根拠
- hidden_choicesがあれば1本はそこから提示する
- ルート名は具体的かつ短く（5-10字）
- 各ルートに「前提条件」と「注意点」を必ず付記
- 末尾の注意書きは必ず付与する

#### ルート名の例
- 良い例: 「転職活動を開始」「副業から始める」「休息と充電」「環境だけ変える」
- 悪い例: 「キャリアチェンジルート」「リスクテイク型」「安全策」

#### ルート選定の優先順位
1. ユーザーが明示した identified_choices から（最優先）
2. 構造化抽出の intended_action に関連するパターンから
3. hidden_choices から1本（推測される未認識の選択肢）

### セクション4: 各ルートの比較表（表形式）

**目的**: 客観的な比較材料を提供する。判断はユーザーに委ねる。

#### テンプレート

```
■ ルート比較

| 項目 | 現状維持 | ルート① | ルート② | ルート③ |
|------|---------|---------|---------|---------|
| 向いている人 | [特性] | [特性] | [特性] | [特性] |
| 主なリスク | [リスク] | [リスク] | [リスク] | [リスク] |
| 必要な行動 | [行動] | [行動] | [行動] | [行動] |
| 必要な期間（目安） | [期間] | [期間] | [期間] | [期間] |
| 前提条件 | [条件] | [条件] | [条件] | [条件] |
| 類似事例数 | N件 | N件 | N件 | N件 |
| 根拠の質 | [観測済み/仮説] | [...] | [...] | [...] |

○ あなたの制約との相性
- [constraintsの各項目との相性を各ルートで評価]
- [risk_toleranceとの相性]
```

#### 記述ルール
- 「成功率」「確率」「的中率」は絶対に入れない
- 類似事例数は正直に記載する（少なければ少ないと書く）
- 「正解」を示唆する表現は使わない
- 「向いている人」は特性として記述し、ユーザーをカテゴリに当てはめない
- 「根拠の質」は事例の credibility_rank とアウトカム観測状況に基づく

#### 「あなたの制約との相性」の書き方
- constraints の各項目をルートごとに評価する
- 例: 「経済的制約: ルート①は初期費用が低いため制約と両立しやすい。ルート②は一時的な収入減少の可能性がある」
- 相性の良し悪しは事実ベースで記述。判断や推奨はしない

#### 軽量版（light）の場合
- 比較表のみ記載。「あなたの制約との相性」は省略
- 列数はルート2本まで（現状維持 + ルート1本）

### セクション5: 今日の1アクション（200-400字）

**目的**: 最小の具体的行動を1つだけ提案する。

#### テンプレート

```
■ 今日の1アクション

[所要時間: 5-15分]

[具体的な行動の提案。「考える」「調べる」ではなく、外部に痕跡が残る行動]

例:
- 「気になるルートについて、1つだけ情報を検索して、スクリーンショットを保存する」
- 「信頼できる人に、今の違和感を3行で伝える」
- 「ルート①の前提条件を1つだけ確認する（電話/メール/検索）」

※ これは小さな実験です。やってみた結果から、
  次に何を感じるかが新しい情報になります。
```

#### 記述ルール
- 具体的であること（5W1Hが分かる）
- 所要時間は15分以内
- 「考える」「振り返る」「整理する」等の内面的行動ではなく、**外部に痕跡が残る行動**
  - 良い例: 検索してスクショ、人にメッセージを送る、電話をかける、ファイルを作成
  - 悪い例: じっくり考える、ノートに気持ちを書く、瞑想する
- 提案は**1つだけ**。複数提案しない
- ルートの中で最もユーザーの関心が高いと思われるものに関連させる
- 「〜してください」ではなく提案形式（「〜してみるのはどうでしょうか」）

### セクション6: 30日/90日実験プラン（400-600字）

**目的**: 小さな実験として次のステップを設計する。

#### テンプレート

```
■ 実験プラン

○ 30日実験
- 仮説: 「[ルート名]に関心があるなら、[具体的な行動]を30日間試すと、
  [期待される情報/変化]が得られる」
- やること: [週1-2回の具体的行動、所要時間付き]
- 観察すること: [何を記録するか]
- 判断基準: 「30日後に[具体的な状態]なら、次のステップに進んでみる」

○ 90日実験（30日実験後に検討）
- 30日実験の結果を踏まえた次のステップの概要
- より踏み込んだ行動の例
- 判断基準: 「90日後に[具体的な状態]なら、本格的に動く」

※ 実験なので「失敗」はありません。
  「やってみた結果、違うと分かった」も貴重な情報です。
```

#### 記述ルール
- 「実験」という位置づけ（失敗OK）を明示する
- 具体的な行動と期間を記載する
- 判断基準を明示する（いつ、何を基準に次を決めるか）
- 「すべきです」「しなければなりません」禁止
- 30日実験は詳細に。90日実験は概要レベルでよい
- 観察すること = 感情や状態の変化ではなく、外部から観測可能な事実
  - 良い例: 「週何回行動したか」「何件の情報を集めたか」「誰と話したか」
  - 悪い例: 「気持ちがどう変わったか」「モチベーションの変化」

## 4. レポート全体の禁止事項

以下の10項目はレポート全体を通じて厳守する。

| # | 禁止事項 | 具体例 |
|---|---------|--------|
| 1 | 断定的予測 | 「〇〇になります」「〇〇でしょう」「〇〇に違いありません」 |
| 2 | 確率・パーセンテージ | 「70%の確率で」「成功率は高い」「半分以上の人が」 |
| 3 | 易経・八卦・卦名 | 「乾」「坤」「六十四卦」「八卦」「爻」 |
| 4 | 占い・スピリチュアル用語 | 「運勢」「星回り」「気の流れ」「波動」「宇宙」 |
| 5 | 「正解」「最適解」「ベスト」 | 「これが正解です」「最適な選択は」「ベストな道は」 |
| 6 | 「おすすめ」「推奨」 | 「おすすめはルート①です」「このルートを推奨します」 |
| 7 | 医療・法律・金融の専門的助言 | 「うつ病の可能性があります」「法的に〇〇すべき」「投資は〇〇」 |
| 8 | 構造化ラベル名 | 「current_state」「energy_direction」「phase_stage」等の技術用語 |
| 9 | タイプ分類・ラベリング | 「あなたは〇〇タイプです」「あなたは内向型です」 |
| 10 | 義務的表現 | 「〜すべき」「〜しなければならない」「〜する必要があります」 |

### 5.2 セクション別文字数（フル版）

| セクション | 文字数目安 |
|-----------|-----------|
| 1. 現在地の構造化 | 800-1,200字 |
| 2. このまま進んだ場合 | 600-1,000字 |
| 3. 別ルート | 800-1,200字 |

| 5. 今日の1アクション | 200-400字 |
| 6. 実験プラン | 400-600字 |

### 5.3 事例引用

| 要件 | 基準 |
|------|------|
| 最低引用数 | 6件（セクション2に3件、セクション3に3件以上） |
| 根拠ラベル | 全事例に必須（「観測済み」or「仮説」） |
| 匿名化 | 全事例に適用済み |
| 類似度閾値 | 高類似(>=0.70)を優先。中類似(0.50-0.70)は仮説ラベル付き |

### 5.4 自動チェック項目

レポート生成後、以下の項目を自動チェックする（内部用、ユーザーには見せない）:

1. 禁止語10項目のスキャン結果
2. 断定表現の検出数
3. 確率・パーセンテージの検出数
4. 根拠ラベルの付与漏れ
5. 各シナリオの最低事例数（3件）の充足
6. 匿名化の適用状況
7. 総文字数
8. 6セクションの完成状況

## 7. 人間レビューのチェックポイント

レポート納品前に、人間レビュアーが以下を確認する:

1. **禁止事項の違反がないか** — 10項目を目視確認
2. **ユーザーの言葉が活かされているか** — 原文の引用が適切に含まれているか
3. **事例が匿名化されているか** — 個人・企業が特定できないか
4. **ルートに優劣がついていないか** — 特定ルートへの誘導がないか
5. **構造化ラベルが露出していないか** — 技術用語が混入していないか
6. **注意書きが付与されているか** — セクション2,3の末尾注意書き
7. **要約合意ゲートの内容と矛盾がないか** — セクション1が合意内容と整合するか
