## 1. フロー概要

```
┌──────────────────────────────────────────────────────────────────────┐
│                        全体フロー                                    │
│                                                                      │
│  ユーザー入力           Safety Filter          ナラティブ収集        │
│  （自由記述）    ──→    （L1/L2/L3）    ──→    （構造化抽出）        │
│       │                     │                       │               │
│       │              CRITICAL/HIGH                   │               │
│       │              → サービス遮断                  │               │
│       │              テンプレート送信                 │               │
│       │                                              │               │
│       │                                    confidence < 0.6?         │
│       │                                      ├── Yes ──→ 追加質問   │
│       │                                      │          （1往復）    │
│       │                                      │              │       │
│       │                                      └── No ───┐    │       │
│       │                                                 │    │       │
│       │                                                 ▼    ▼       │
│       │                                           要約合意ゲート     │
│       │                                        「合っている」？      │
│       │                                         ├── Yes ──→ レポート │
│       │                                         │           生成     │
│       │                                         └── No ──→ 修正    │
│       │                                                    再抽出    │
└──────────────────────────────────────────────────────────────────────┘
```

### ステップ一覧

| Step | 名称 | 目的 | 最大往復数 |
|------|------|------|-----------|
| 1 | ナラティブ入力 | 違和感を自由に語ってもらう | — |
| 2 | Safety Filter | 危機ケースの検知・遮断 | — |
| 3 | LLM構造化抽出 | 自然言語 → 10項目の構造化ラベル | — |
| 4 | フォールバック質問 | confidence < 0.6 の項目を補完 | 1往復 |
| 5 | 要約合意ゲート | 抽出結果の自然言語要約に対する合意取得 | 1往復 |
| — | レポート生成 | 6セクションレポートの作成・人間レビュー・納品 | — |

**最大やりとり回数**: 3往復（ナラティブ + 追加質問 + 要約合意）

## 2. Step 1: ナラティブ入力（自由記述）

### 目的

ユーザーに「違和感」を自分の言葉で語ってもらう。
構造化質問（選択肢式）は使わない。LLMが構造化を担当するため、ユーザーには自由に書いてもらう。

### 入力ガイド（ユーザーに送る文面）

```
【あなたの状況を教えてください】

今の状況を、あなたの言葉で自由に書いてください。
きれいにまとめる必要はありません。思いつくままで大丈夫です。
（目安: 200文字以上あると、より正確な整理ができます）

もし何を書けばいいか迷ったら、以下をヒントにしてみてください:

・最近、ふとした瞬間に頭をよぎることは？
・外からは順調に見えるけど、自分の中では違うと感じることは？
・もし何の制約もなかったら、何を変えたい？

上のヒントに答える形でもいいですし、
全く関係ないことを書いてもらっても構いません。
あなたの感じていることを、そのまま教えてください。
```

### 設計意図

- **補助質問3つは「種」**。ユーザーの筆を動かすためのものであり、3つ全てに答える必要はない
- **200文字の目安**を示すことで、「一言で終わる」入力を防ぐ。ただし強制はしない
- **「きれいにまとめなくていい」**と明示することで、構造化のプレッシャーを外す
- ユーザーが書いたテキストがStep 3（LLM構造化抽出）の唯一の入力となる

### 入力が短すぎた場合（100文字未満）

追加の引き出し質問を1つだけ送る:

```
ありがとうございます。
もう少し詳しく聞かせていただけると、より正確な整理ができます。

今の状況について、もう少し具体的に教えていただけますか？
たとえば、いつ頃からそう感じ始めたか、きっかけがあったかなど。
```

## 3. Step 2: Safety Filter

### 概要

入力テキストに対して `docs/safety_design.md` の3層検知アーキテクチャを適用する。

### 判定フロー

```
入力テキスト
    │
    ▼
Layer 1: キーワード検知（即時・自動）
    │
    ├── フラグあり → Layer 2 へ
    └── フラグなし → Layer 2 へ（全入力に対して実行）
    │
    ▼
Layer 2: LLM分類（文脈理解・自動）
    │
    ├── CRITICAL → 即時遮断（緊急テンプレート送信）→ Layer 3 人間確認
    ├── HIGH     → サービス一時停止（専門窓口案内）→ Layer 3 人間確認
    ├── MEDIUM   → 注意フラグ付きで Step 3 へ継続
    ├── LOW      → 通常処理（Step 3 へ）
    └── FALSE_POSITIVE → 通常処理（Step 3 へ、ログに記録）
```

### CRITICAL/HIGH 時の対応

- サービス遮断。レポート生成は行わない
- `docs/safety_design.md` セクション2のテンプレートを送信
- 人間（サービス提供者）に通知し、最終判断を仰ぐ
- 返金対応を行う

### MEDIUM 時の対応

- サービスは継続するが、レポートに注意フラグを付与
- レポートの冒頭に免責事項を追加:
  ```
  当サービスは意思決定の整理をお手伝いするもので、
  医療・心理・法律の専門的な助言は行っておりません。
  必要に応じて、専門家にもご相談ください。
  ```
- 人間レビュー時に注意フラグを確認

## 4. Step 3: LLM構造化抽出

### 概要

ナラティブテキストを `prompts/extraction_prompt_individual.md`（個人特化版、別途作成）で構造化する。

### 抽出項目（10項目）

既存の `prompts/structured_extraction.md` の7項目をベースに、個人向けの新規3項目を追加。

#### 既存項目（5項目 + domain固定 + confidence）

| # | 項目 | 選択肢数 | 参照元 |
|---|------|---------|--------|
| 1 | current_state（現在の状態） | 15 | `structured_extraction.md` |
| 2 | energy_direction（エネルギーの方向） | 6 | `structured_extraction.md` |
| 3 | intended_action（意図する行動） | 22 | `structured_extraction.md` |
| 4 | trigger_nature（トリガーの性質） | 8 | `structured_extraction.md` |
| 5 | phase_stage（フェーズ段階） | 6 | `structured_extraction.md` |
| — | domain | 「個人」固定 | — |
| — | confidence | 各項目に0.0-1.0 | `structured_extraction.md` |

#### 新規追加項目（3項目）

| # | 項目 | 定義 | 選択肢 |
|---|------|------|--------|
| 6 | constraints（制約条件） | 変化を阻む制約 | 複数選択可 |
| 7 | risk_tolerance（リスク許容度） | リスクへの姿勢 | 5段階 + 不可逆性フラグ |
| 8 | choice_set（選択肢集合） | 本人が認識している選択肢 | 自由リスト |

##### constraints（制約条件）の選択肢

| 制約 | 典型的なキーワード |
|------|------------------|
| お金 | 貯金がない、ローンがある、収入が減る、経済的に不安 |
| 時間 | 忙しい、時間がない、子どもが小さい、介護がある |
| 家族の義務 | 親の面倒、配偶者の反対、子どもの学校、家族の期待 |
| 健康 | 体調が悪い、持病がある、体力に不安 |
| 地理的制約 | 引っ越せない、地方にいる、海外にいる |
| スキル不足 | 経験がない、資格がない、自信がない |
| 人間関係 | 上司との関係、職場の人間関係、パートナーとの関係 |
| 年齢 | もう遅い、若すぎる、年齢制限 |
| 情報不足 | 何から調べればいいか分からない、選択肢が見えない |
| 制約なし | 特に制約を感じていない |

複数選択可。テキストから読み取れる制約を全て抽出する。

##### risk_tolerance（リスク許容度）の定義

| レベル | 定義 | 典型的な判断基準 |
|--------|------|-----------------|
| 1: 回避 | リスクを極力避けたい。現状維持が安全 | 「失敗したくない」「確実なものだけ」 |
| 2: 慎重 | 小さなリスクなら取れる。大きな変化は怖い | 「少しずつなら」「副業ならいいけど」 |
| 3: 中立 | リスクとリターンを天秤にかけて判断 | 「損得を考えて」「合理的に判断したい」 |
| 4: 許容 | ある程度のリスクは覚悟の上 | 「多少の犠牲は仕方ない」「挑戦したい」 |
| 5: 積極 | 大きなリスクも取れる。失うものは少ない | 「全部変えてもいい」「もう怖くない」 |

**不可逆性フラグ**:
- `true`: 取ろうとしている行動が不可逆的（退職、離婚、移住など）
- `false`: 取ろうとしている行動が可逆的（副業開始、学び直し、転職活動開始など）
- `unknown`: テキストから判断できない

##### choice_set（選択肢集合）の抽出ルール

- ユーザーが言及した選択肢を全てリスト化する
- 各選択肢に「本人の傾き度合い」を 1-5 で付ける（1: 消極的、5: 強く傾いている）
- 選択肢が明確に見えていない場合は `"status": "unclear"` とする
- 例:
  ```json
  {
    "options": [
      {"label": "今の会社に残る", "lean": 2},
      {"label": "転職する", "lean": 4},
      {"label": "独立する", "lean": 3}
    ],
    "status": "partial"
  }
  ```
- status の値:
  - `"clear"`: 選択肢が明確に見えている
  - `"partial"`: 一部見えているが他にもありそう
  - `"unclear"`: 選択肢自体が見えていない（「何を変えればいいか分からない」）

### 出力形式

```json
{
  "current_state": {
    "primary": "15の選択肢から1つ",
    "confidence": 0.0-1.0,
    "secondary": "該当する場合",
    "secondary_confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "energy_direction": {
    "primary": "6つの選択肢から1つ",
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "intended_action": {
    "primary": "22の選択肢から1つ",
    "confidence": 0.0-1.0,
    "secondary": "該当する場合",
    "secondary_confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "trigger_nature": {
    "primary": "8つの選択肢から1つ",
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "phase_stage": {
    "primary": "6つの選択肢から1つ",
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "domain": "個人",
  "constraints": {
    "items": ["お金", "家族の義務"],
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "risk_tolerance": {
    "level": 1-5,
    "irreversibility": true|false|"unknown",
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "choice_set": {
    "options": [
      {"label": "選択肢の内容", "lean": 1-5}
    ],
    "status": "clear|partial|unclear",
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  }
}
```

## 5. Step 4: フォールバック質問（confidence < 0.6 の項目がある場合）

### トリガー条件

構造化抽出の結果、confidence が 0.6 未満の項目が1つ以上ある場合に発動。

### 設計方針

- 選択肢を見せて選んでもらう形式は使わない
- 追加のナラティブを引き出す自然言語の質問を投げる
- **1回のメッセージでまとめて質問する**（最大3項目まで。4項目以上は優先度の高い3つに絞る）
- **1往復で完結させる**（追加質問の追加質問はしない）

### 項目別フォールバック質問テンプレート

#### current_state が不明確（confidence < 0.6）

```
今の生活を10点満点で表すと何点ですか？
その理由を教えてください。
```

**設計意図**: 点数化によって「安定・順調」と「停滞・閉塞」の区別がつきやすくなる。理由を求めることで根拠も得られる。

#### energy_direction が不明確（confidence < 0.6）

```
今のあなたは、外に向かって動きたい感じですか？
それとも、立ち止まって内側を整理したい感じですか？
```

**設計意図**: 内向/外向の大きな方向性を直感的に掴む。

#### intended_action が不明確（confidence < 0.6）

```
今、一番やりたいこと（またはやらなきゃと思っていること）は
何ですか？まだ迷っている段階でも構いません。
```

**設計意図**: 「やりたいこと」と「やらなきゃと思っていること」の両方を聞くことで、意図と義務の区別がつく。

#### trigger_nature が不明確（confidence < 0.6）

```
この違和感はいつ頃から感じ始めましたか？
きっかけになった出来事はありましたか？
```

**設計意図**: 時期ときっかけを聞くことで、突発的外圧/漸進的変化/内発的衝動の区別がつく。

#### phase_stage が不明確（confidence < 0.6）

```
この状況は、始まったばかりですか？
それとも、もうずっと続いていますか？
```

**設計意図**: 時間経過の感覚を聞くことで、始まり/展開中期/終結などの区別がつく。

#### constraints が不明確（confidence < 0.6）

```
変化を考えるとき、一番気になる制約は何ですか？
（お金、時間、家族、健康、場所など、なんでも）
```

**設計意図**: 「なんでも」と付けることで、リストにない制約も引き出せる。

#### risk_tolerance が不明確（confidence < 0.6）

```
もし転職や独立をするとして、
収入が半分になっても大丈夫だと感じますか？
```

**設計意図**: 具体的な数字（半分）を出すことで、抽象的な「リスク許容度」を体感的に判断できる。

#### choice_set が不明確（confidence < 0.6）

```
今、頭の中にある選択肢を全部書き出してみてください。
まとまっていなくてOKです。
「何も思いつかない」も立派な答えです。
```

**設計意図**: 「まとまっていなくてOK」と明示することで書き出しのハードルを下げる。「何も思いつかない」も受け入れることで、choice_set の status が `"unclear"` であることを確認できる。

### フォールバック質問の統合例

confidence < 0.6 の項目が trigger_nature、constraints、choice_set の3つだった場合:

```
もう少し教えていただけると、より正確な整理ができます。

① この違和感はいつ頃から感じ始めましたか？
  きっかけになった出来事はありましたか？

② 変化を考えるとき、一番気になる制約は何ですか？
  （お金、時間、家族、健康、場所など）

③ 今、頭の中にある選択肢を全部書き出してみてください。
  まとまっていなくてOKです。
```

### 優先順位（4項目以上が不明確な場合の絞り込み）

1. current_state（レポートの土台）
2. choice_set（シナリオ分岐に直結）
3. constraints（シナリオの現実性に直結）
4. trigger_nature（現在地の文脈理解に必要）
5. risk_tolerance（ルート比較に必要）
6. intended_action
7. energy_direction
8. phase_stage

## 6. Step 5: 要約合意ゲート（Summary Agreement Gate）

### 目的

レポート生成前の最重要ステップ。
構造化抽出の結果が「ユーザーの認識」と合っているかを確認する。

ここでズレたままレポートを作ると、「的外れなレポート」になり、信頼を失う。
逆に、ここで合意が取れれば、レポートへの納得感が大幅に上がる。

### 要約変換ルール

構造化抽出結果（JSON）を自然言語で5-8行の要約に変換する。
以下のテンプレートを使用するが、不自然にならないよう文脈に合わせて調整する。

### 要約テンプレート

```
【あなたの状況の要約】

現在の状況: [current_stateの自然言語化]
変化のきっかけ: [trigger_natureの自然言語化]
今考えていること: [intended_actionの自然言語化]
主な制約: [constraintsの自然言語化]
リスクへの姿勢: [risk_toleranceの自然言語化]
見えている選択肢: [choice_setの自然言語化]

この理解で合っていますか？
→ 合っている / 違う部分がある / 補足したい
```

### 自然言語化の例

| 項目 | 構造化ラベル | 自然言語化の例 |
|------|------------|---------------|
| current_state | 停滞・閉塞 | 「大きな問題はないけれど、ここ数年同じことの繰り返しで、先が見えない感覚がある」 |
| current_state | 安定・順調 | 「生活は安定していて、外から見ると特に問題はない状況」 |
| trigger_nature | 内発的衝動 | 「はっきりしたきっかけがあったというより、内側から湧いてきた違和感」 |
| trigger_nature | 関係性変化 | 「身近な人との関係の変化がきっかけになっている」 |
| intended_action | 選択・判断 | 「いくつかの選択肢の中で、どれを選ぶべきか考えている段階」 |
| constraints | お金, 家族の義務 | 「経済面と家族への責任が、動きにくさの主な要因」 |
| risk_tolerance | 2: 慎重 | 「大きな変化には慎重で、小さく試せるものから始めたい」 |
| choice_set | partial | 「転職と独立が頭にあるが、他にもありそうな感覚がある」 |

### 自然言語化の禁止事項

- 易経・八卦の用語を使わない
- 構造化ラベルの名称をそのまま見せない（「current_state は停滞・閉塞です」は禁止）
- 「診断結果」「予測」「正解」の表現を使わない
- 断定的な表現を避ける（「あなたは〇〇です」ではなく「〇〇と感じている状況」）

### 合意後の分岐

| ユーザーの回答 | アクション |
|-------------|----------|
| 「合っている」 | レポート生成へ進む |
| 「違う部分がある」 | 違う部分を聞き、該当項目を再抽出。修正後に再度要約を提示 |
| 「補足したい」 | 追加テキストを受け取り、全体を再抽出。再度要約を提示 |

### 再提示は1回まで

- 修正・補足後の再要約で再び「違う」と言われた場合は、人間（サービス提供者）が介入する
- 「お伝えいただいた内容をもとに、担当者が直接確認させていただきます」と返し、手動対応に切り替える

## 7. ココナラ手動運用時のフロー

### Phase 3 MVP: ココナラでの手動運用

ココナラの「悩み相談・カウンセリング」カテゴリで出品。
メッセージ機能を使ったやりとりで完結する。

### やりとりの流れ

```
【購入後のやりとり】

往復1: ナラティブ収集
  出品者 → 購入者: 入力ガイド送信（Step 1 のテンプレート）
  購入者 → 出品者: ナラティブ記述（自由記述）

（裏方: Safety Filter → LLM構造化抽出）

往復2: 追加質問（該当時のみ）
  出品者 → 購入者: フォールバック質問（confidence < 0.6 の項目）
  購入者 → 出品者: 追加の記述
  ※ confidence が全項目 0.6 以上なら、この往復はスキップ

往復3: 要約合意
  出品者 → 購入者: 要約テンプレート送信
  購入者 → 出品者: 「合っている」/「違う部分がある」/「補足したい」

（裏方: レポート生成 → 人間レビュー）

納品:
  出品者 → 購入者: 6セクションレポート送信
```

### 各ステップの実施者

| ステップ | 実施者 | ツール |
|---------|--------|--------|
| 入力ガイド送信 | 人間 | ココナラメッセージ（テンプレートコピペ） |
| Safety Filter (L1) | LLM + 人間確認 | キーワード検索 + LLM API |
| Safety Filter (L2) | LLM | LLM API（分類プロンプト） |
| Safety Filter (L3) | 人間 | CRITICAL/HIGH 時のみ |
| 構造化抽出 | LLM | `extraction_prompt_individual.md` |
| フォールバック質問作成 | LLM + 人間確認 | テンプレートベース |
| 要約作成 | LLM + 人間確認 | 要約テンプレート |
| レポート生成 | LLM | レポート生成プロンプト（別途設計） |
| レポートレビュー | 人間 | 目視確認・修正 |
| 納品 | 人間 | ココナラメッセージ |

### 時間の目安

| フェーズ | 所要時間 |
|---------|---------|
| ナラティブ受信〜構造化抽出 | 10分 |
| フォールバック質問作成（該当時） | 5分 |
| 要約作成・送信 | 5分 |
| レポート生成 | 15分 |
| 人間レビュー・修正 | 15-30分 |
| **合計** | **55分**（`docs/pricing_cost_structure.md` と整合） |

### 納品目安

- **入金確認後48時間以内**
- ユーザーからの返信待ち時間は除く
- CRITICAL/HIGH 判定時は返金対応（レポート生成しない）

### ココナラ出品時の注意

- 商品説明に免責事項を記載する（`docs/safety_design.md` セクション4参照）
- 「意思決定の整理をお手伝いするサービス」と明示する
- 「予測」「診断」「占い」の表現は避ける
- 「レポートは参考情報であり、最終判断はご本人にあります」と明記する

## 8. 品質ゲート

### Phase 1 完了の品質ゲート

Phase 1（質問フロー設計）を完了とみなすための基準。
自作ペルソナ10ケースでの壁打ちテストを通過すること。

| 基準 | 閾値 | 測定方法 |
|------|------|---------|
| Safety Filter の偽陰性 | 0件 | 危機ケース（自作3件以上）に対して全件検知 |
| Safety Filter の偽陽性率 | 20%以下 | 非危機ケースに対する誤検知率 |
| 構造化抽出の confidence 平均 | >= 0.65 | 10ケースの全項目の confidence 平均 |
| confidence < 0.6 の出現率 | 40%以下 | 10ケース × 8項目 = 80判定のうち、0.6未満の割合 |
| 要約合意率 | >= 80% | 初回要約で「合っている」の割合（10ケース中8件以上） |
| フロー完走率 | 100% | 10ケース全てが3往復以内でレポート生成まで到達 |

### テストケースの設計

10ケースは以下のバリエーションをカバーする:

| # | ペルソナ | 主な違和感 | 狙う抽出パターン |
|---|---------|-----------|-----------------|
| 1 | 30代会社員・男性 | キャリアの停滞感 | 停滞・閉塞 / 内発的衝動 |
| 2 | 40代管理職・女性 | ワークライフバランス | 安定・順調 / 循環・往復 |
| 3 | 20代フリーター・男性 | 方向性が見えない | 未知・探索 / 停止 |
| 4 | 50代会社員・男性 | 定年後の不安 | 衰退・下降 / 漸進的変化 |
| 5 | 30代主婦・女性 | 社会復帰への迷い | 依存・束縛 / 関係性変化 |
| 6 | 20代起業家・男性 | 事業の岐路 | 転換期 / 選択・判断 |
| 7 | 40代転職検討中・男性 | 安定か挑戦か | 忍耐・持久 / 慎重・観察 |
| 8 | 30代・自殺念慮あり（Safety テスト） | 危機ケース | CRITICAL判定 |
| 9 | 40代・DV被害示唆（Safety テスト） | 危機ケース | HIGH判定 |
| 10 | 30代・軽度の抑うつ言及（Safety テスト） | 注意ケース | MEDIUM判定 |

### テスト実行手順

1. 各ペルソナのナラティブテキスト（200-500文字）を作成
2. Safety Filter を実行し、判定結果を記録
3. 構造化抽出を実行し、JSON出力を記録
4. confidence < 0.6 の項目に対してフォールバック質問を生成
5. フォールバック回答（自作）を追加入力し、再抽出
6. 要約を生成し、ペルソナの意図と照合
7. 全結果をスプレッドシートに記録し、品質ゲートの数値を算出

## 注意事項（全体を通じて）

### 用語制限

| 使ってはいけない用語 | 代替表現 |
|-------------------|---------|
| 易経、八卦、卦 | （使わない。内部処理の話をしない） |
| 予測 | 「類似事例からのパターン」「シナリオ」 |
| 診断 | 「整理」「現在地の把握」 |
| 正解 | 「あなたに合った方向」「選択肢の比較」 |
| 占い | （使わない） |
| 治す、治療 | （使わない。スコープ外） |

### トーン

- 全体を通じて「意思決定の整理をお手伝い」のトーン
- 上から目線の断定を避ける
- 「あなたの状況を整理すると」「類似の事例ではこういうパターンがあります」
- 共感はするが、過度な寄り添いはしない（カウンセリングではない）

### 構造化の原則

- **ユーザーには自然言語で問いかける**（構造化質問は使わない）
- **構造化はLLMが裏方でやる**（ユーザーには構造化ラベルを見せない）
- **要約は自然言語で返す**（JSONを見せない）
- ユーザーが触れるのは常に自然言語だけ
