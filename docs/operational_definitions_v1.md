## 1. 爻位置の一般定義（6段階モデル）

### 1.1 各爻の基本定義

| 爻 | 段階名 | 定義 | テキスト証拠条件 | 典型的な期間 |
|----|--------|------|------------------|--------------|
| **初爻（1）** | 萌芽・潜伏 | 変化の芽が生じているが、まだ表面化していない。外部からは見えにくい。資源・認知・準備が不十分。 | 「構想段階」「創業初期」「アイデア段階」「設立準備」「R&D初期」「調査中」「検討開始」 | 0-2年目 |
| **二爻（2）** | 展開・基盤 | 変化が外部から認識され始めた。基盤が形成され、軌道に乗り始めている。協力者が現れる。 | 「軌道に乗る」「PMF達成」「黒字化」「初の大型契約」「基盤確立」「チーム形成」「市場認知」 | 2-5年目 |
| **三爻（3）** | 分岐・危機 | 内部と外部の境界に位置する。岐路に立ち、選択を迫られる。危険と機会が同居する不安定な状態。 | 「転換点」「岐路」「選択肢」「リスク顕在化」「競合激化」「決断を迫られる」「分社化検討」 | 規模・状況依存 |
| **四爻（4）** | 接近・準備 | 頂点に近づいている。準備は整ったが、まだ主導権は握っていない。慎重さと根回しが求められる。 | 「次期候補」「M&A検討」「IPO準備」「上位との調整」「承認待ち」「基盤完成」「拡大期」 | 規模・状況依存 |
| **五爻（5）** | 頂点・主導 | 最も力がある位置。リーダーシップを発揮すべき時。決断と実行が求められる。責任も最大。 | 「業界首位」「決定権」「経営判断」「主導」「リーダー」「CEO決断」「全社戦略」「頂点」 | 規模・状況依存 |
| **上爻（6）** | 極限・過剰 | 行き過ぎた状態。ピークを過ぎて下降に向かう、または限界に達している。次の変化への転換期。 | 「行き過ぎ」「限界」「成熟市場」「引退」「後継問題」「極端」「過剰拡大」「バブル」「終末」 | 終盤・転換期 |

### 1.2 爻の判定フローチャート

```
開始
  │
  ├─ 外部から認識可能か？
  │     NO → 初爻（1）
  │     YES ↓
  │
  ├─ 基盤・軌道は確立しているか？
  │     NO → 二爻（2）の前半
  │     YES ↓
  │
  ├─ 岐路・選択を迫られているか？
  │     YES → 三爻（3）
  │     NO ↓
  │
  ├─ 主導権を握っているか？
  │     NO → 四爻（4）：接近中
  │     YES ↓
  │
  ├─ ピーク/極限に達しているか？
  │     NO → 五爻（5）：頂点
  │     YES → 上爻（6）：過剰/転換
```

#### 企業（company）

## 2. 採択条件

### 2.1 卦マッピングの必須条件

事例をある卦にマッピングするには、以下の**全て**を満たすこと：

| # | 条件 | 検証方法 |
|---|------|---------|
| 1 | **テキスト根拠の存在** | story_summaryまたはソース文書に、卦の意味に対応する記述が存在する |
| 2 | **爻辞との対応** | 選択した爻の爻辞（原文）が事例の状況に対応する説明が可能 |
| 3 | **候補卦の比較検討** | 最低3つの候補卦を検討し、選択理由と棄却理由を記録 |
| 4 | **反証の検討** | 「この卦ではない」理由を説明できる |

### 2.2 爻位置マッピングの必須条件

| # | 条件 | 検証方法 |
|---|------|---------|
| 1 | **段階の明確性** | 事例がどの発展段階にあるか、1.1の定義に基づき判定できる |
| 2 | **テキスト証拠** | 1.1の「テキスト証拠条件」に該当する語句がソースに存在する |
| 3 | **時点の特定** | どの時点（発表時点/実行時点/結果確定時点）での判定かが明確 |

### 2.3 採択チェックリスト

```
□ 八卦（trigger_hex, action_hex）は2.4の定義に基づいて選択したか
□ 64卦は上卦・下卦の組み合わせとして論理的に正しいか
□ 爻位置は1.1の定義に基づいて選択したか
□ 爻辞の原文を確認し、対応説明を記載したか
□ 3つ以上の候補卦を検討したか
□ 棄却した卦の理由を記載したか
```

### 2.4 八卦の採択条件（テキスト証拠リスト）

以下のテキスト証拠が見られた場合、対応する八卦を採択候補とする：

#### 乾（天・創造・主導）
- **採択**: 「創業」「新規参入」「主導」「リーダーシップ」「大胆な投資」「拡大戦略」「攻めの経営」「先頭に立つ」「イニシアチブ」「トップダウン」
- **反例**: 受動的対応、他社追随、現状維持

#### 坤（地・受容・支援）
- **採択**: 「受け入れる」「支える」「地道」「継続」「フォロワー」「サポート」「基盤」「忍耐」「下支え」「従う」
- **反例**: 主導的行動、急激な変化、対立姿勢

#### 震（雷・衝撃・始動）
- **採択**: 「突然」「衝撃」「スキャンダル」「発覚」「急展開」「緊急」「ショック」「突発」「起爆剤」「覚醒」
- **反例**: 予定通り、漸進的、計画的変化

#### 巽（風・浸透・適応）
- **採択**: 「浸透」「根回し」「交渉」「調整」「徐々に」「信用」「口コミ」「ネットワーク」「柔軟」「適応」
- **反例**: 強引な手法、即効性、直接対決

#### 坎（水・危険・困難）
- **採択**: 「危機」「困難」「リスク」「問題」「障害」「苦境」「試練」「損失」「赤字」「訴訟」「不正」
- **反例**: 順風満帆、安定期、成功局面

#### 離（火・明晰・分離）
- **採択**: 「発見」「可視化」「公開」「透明化」「分析」「判断」「分離」「独立」「炎上」「注目」「上場」
- **反例**: 隠蔽、統合、水面下の動き

#### 艮（山・停止・蓄積）
- **採択**: 「停止」「待機」「蓄積」「守り」「ブレーキ」「撤退」「縮小」「現状維持」「防衛」「境界」
- **反例**: 積極的拡大、攻勢、変化の推進

#### 兌（沢・喜び・交流）
- **採択**: 「喜び」「利益」「収益」「対話」「合意」「サービス」「顧客満足」「エンタメ」「和解」「提携」
- **反例**: 対立、損失、孤立、内向き

## 3. 棄却条件

### 3.1 絶対棄却条件（これに該当したら必ず棄却）

| # | 棄却条件 | 理由 |
|---|---------|------|
| 1 | **テキスト根拠なし** | 「何となくそう感じた」では採択不可 |
| 2 | **爻辞との不整合** | 選択した爻の爻辞と事例が矛盾する |
| 3 | **時系列の矛盾** | 結果確定後の視点で「初爻」を割り当てる等 |
| 4 | **同一事例の重複** | 既存事例と target_name + period が同一 |
| 5 | **架空・推測事例** | 実在しない事例、または「もし〜だったら」の仮定 |

### 3.2 相対棄却条件（より適切な代替がある場合に棄却）

| # | 棄却条件 | 確認方法 |
|---|---------|---------|
| 1 | **より適合する卦が存在** | 候補卦の fit_score 比較で明確に劣る |
| 2 | **爻位置の不明確さ** | 複数の爻に同等に該当する（段階が特定できない） |
| 3 | **視点の不統一** | 同一事例で主語（主体）が途中で変わる |

### 3.3 典型的な誤マッピングパターン（反例集）

#### パターン1: 結果の後付け解釈

- **誤**: 「倒産したから坎（危機）」
- **正**: 倒産という結果ではなく、その時点での状況・行動を判定すべき
- **対処**: before_state/trigger/action/after の各フェーズを分離して判定

#### パターン2: 抽象的すぎるマッピング

- **誤**: 「変化があったから震」
- **正**: 「突発的」「予期せぬ」「衝撃的」といった具体的特徴が必要
- **対処**: 2.4のテキスト証拠リストに該当する語句を特定

#### パターン3: 爻位置の固定観念

- **誤**: 「創業者だから常に初爻」
- **正**: 創業者でも業界リーダーになれば五爻、衰退期なら上爻
- **対処**: 事例の「時点」での段階を判定

#### パターン4: 単一視点への固執

- **誤**: 企業の視点のみで判定
- **正**: 同一事件でも、企業/従業員/顧客/社会で異なる卦になり得る
- **対処**: 視点を明示し、一貫させる

#### パターン5: 過度な簡略化

- **誤**: 「成功=乾、失敗=坎」
- **正**: 八卦は「性質」であり、「結果」ではない
- **対処**: 行動の性質（攻め/守り/柔軟等）に基づいて判定

## 4. 多段階事象の扱い

### 4.1 ラベル付与の時点定義

多段階で進行する事象（M&A、大型プロジェクト等）は、以下の時点で分割してラベル付与する：

| 時点 | 定義 | 典型的なイベント |
|------|------|-----------------|
| **T0: 発表時点** | 意思決定が外部に公表された時点 | プレスリリース、決算発表、報道 |
| **T1: 実行時点** | 主要な行動が実行された時点 | 契約締結、合併完了、サービス開始 |
| **T2: 結果確定時点** | 結果が明確になった時点 | 1年後の業績、統合完了、撤退完了 |

### 4.2 位相（phase）の定義

```json
{
  "phase": "T0",           // T0/T1/T2
  "phase_label": "発表時点",
  "phase_date": "2020-03-15",
  "phase_description": "M&A発表、株価急騰"
}
```

### 4.3 多段階事象の分割ルール

#### 原則: 1事例 = 1時点

同一事象でも時点が異なれば別事例として登録する。

#### 例: M&A事例の分割

| 時点 | transition_id | 卦 | 爻 | 説明 |
|------|---------------|----|----|------|
| T0 | CORP_JP_100_T0 | 離（発表・可視化） | 4（準備完了） | M&A発表時点 |
| T1 | CORP_JP_100_T1 | 兌（合意・提携） | 5（主導・実行） | 統合実行時点 |
| T2 | CORP_JP_100_T2 | 坤（基盤・継続） | 2（基盤形成） | 統合後の安定期 |

### 4.4 位相間の関係記録

```json
{
  "phase_relation": {
    "parent_event": "ソフトバンク・Sprint合併",
    "phase_sequence": ["CORP_JP_100_T0", "CORP_JP_100_T1", "CORP_JP_100_T2"],
    "current_phase": "T1"
  }
}
```

### 4.5 位相判定の優先順位

複数の時点が混在する記述の場合：

1. **明示的な時点があれば採用**: 「発表時点では」「合併完了後」等
2. **動詞の時制で判定**: 「〜する予定」(T0) / 「〜した」(T1-T2)
3. **評価の確定度で判定**: 「成功した」(T2) / 「期待される」(T0-T1)
4. **不明な場合は T1（実行時点）をデフォルト**

## 5. 八卦マッピングの運用定義

### 5.1 各フェーズでの八卦の役割

| フィールド | 役割 | 何を判定するか |
|-----------|------|---------------|
| before_hex | 初期状態の性質 | 事象開始前の組織/個人の状態 |
| trigger_hex | きっかけの性質 | 変化を引き起こした要因の性質 |
| action_hex | 対応行動の性質 | 主体が取った行動の性質 |
| after_hex | 結果状態の性質 | 変化後の状態の性質 |

### 5.2 八卦選択の操作的手順

```
Step 1: テキストからキーワードを抽出
       ↓
Step 2: 2.4のテキスト証拠リストと照合
       ↓
Step 3: 該当する八卦を候補として列挙
       ↓
Step 4: 候補が複数ある場合、最も本質的な性質を選択
       ↓
Step 5: 選択理由と棄却理由を記録
```

### 5.3 八卦の競合解決ルール

複数の八卦に該当する場合の優先順位：

| 状況 | 解決ルール |
|------|-----------|
| 行動の性質が複合 | 主たる性質（最も影響が大きい）を選択 |
| 同程度に該当 | 易経の伝統的解釈を参照 |
| 判断不能 | 両方を候補として記録し、複数解釈として保存 |

## 6. 64卦の選択基準

### 6.1 上卦・下卦の決定

64卦は上卦（外卦）と下卦（内卦）の組み合わせで決まる。

| 要素 | 役割 | 判定基準 |
|------|------|---------|
| 下卦（内卦） | 内部状態、基盤、自己 | before_hex または主体の本質的性質 |
| 上卦（外卦） | 外部環境、目標、他者 | trigger_hex または外部からの影響 |

### 6.2 64卦選択の操作的手順

```
Step 1: trigger_hex を上卦候補とする
Step 2: before_hex を下卦候補とする
Step 3: 上卦×下卦で64卦を特定
Step 4: 特定した卦の卦辞・意味を確認
Step 5: 事例との整合性を検証
Step 6: 不整合があれば、action_hex を考慮して再検討
```

## 7. 品質保証プロセス

### 7.1 事前チェック（インポート前）

```python
def validate_mapping(case):
    errors = []

    # 1. 必須フィールド存在チェック
    required = ['trigger_hex', 'action_hex', 'story_summary']
    for field in required:
        if not case.get(field):
            errors.append(f"必須フィールド欠落: {field}")

    # 2. 八卦値の妥当性
    valid_hex = ['乾', '坤', '震', '巽', '坎', '離', '艮', '兌']
    for field in ['before_hex', 'trigger_hex', 'action_hex', 'after_hex']:
        if case.get(field) and case[field] not in valid_hex:
            errors.append(f"無効な八卦値: {field}={case[field]}")

    # 3. 爻位置の範囲チェック
    yao = case.get('yao_analysis', {}).get('before_yao_position')
    if yao and not (1 <= yao <= 6):
        errors.append(f"爻位置範囲外: {yao}")

    # 4. テキスト根拠の存在チェック
    if not case.get('logic_memo'):
        errors.append("logic_memo（判定根拠）なし")

    return errors
```

### 7.2 事後チェック（定期監査）

| チェック項目 | 頻度 | 合格基準 |
