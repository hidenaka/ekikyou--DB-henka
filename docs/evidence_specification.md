# 根拠仕様書（Evidence Specification）

**作成日**: 2026-03-09
**ステータス**: 確定（Phase 0）
**目的**: 「類似事例からのシナリオ提示」が根拠として成立するための仕様定義

## 1. 類似の定義

### 1.1 類似度算出に使用する特徴量
以下の特徴量の加重組み合わせで類似度を算出する:

| 特徴量 | 重み | 算出方法 |
|--------|------|---------|
| before_state（初期状態） | 0.25 | 完全一致=1.0, 「・」区切り部分一致=0.5 |
| scale（スケール） | 0.20 | 同一=1.0, 隣接=0.7, 遠い=0.3 |
| trigger_type（変化のきっかけ） | 0.15 | 完全一致=1.0, 部分一致=0.5 |
| action_type（行動タイプ） | 0.15 | 完全一致=1.0, 部分一致=0.5 |
| main_domain（ドメイン） | 0.10 | 完全一致=1.0, 同カテゴリ=0.5 |
| period（時期） | 0.05 | 10年以内=1.0, 20年以内=0.7, それ以上=0.3 |
| duration_years（期間） | 0.05 | 差分5年以内=1.0, 10年以内=0.7, それ以上=0.3 |
| reversibility_score | 0.05 | 差分1以内=1.0, 2以内=0.7, それ以上=0.3 |

### 1.2 類似度の閾値
- **高類似**: スコア >= 0.70（レポートに「類似事例」として提示可能）
- **中類似**: 0.50 <= スコア < 0.70（「参考事例」として提示可能、仮説ラベル付き）
- **低類似**: スコア < 0.50（提示しない）

### 1.3 最低事例数
- 各シナリオの根拠として最低3件の高類似/中類似事例が必要
- 3件未満の場合、そのシナリオには「**事例数不足: 仮説としてのみ提示**」ラベルを付与

## 2. アウトカムの定義

### 2.1 アウトカム分類
既存DBの outcome フィールドを使用:
- **Success**: 目標達成または状況改善が確認された
- **Failure**: 目標未達成または状況悪化が確認された
- **Mixed**: 一部成功・一部失敗
- **PartialSuccess**: 部分的な成功

### 2.2 アウトカム観測の信頼度
既存DBの credibility_rank を活用:
- **S/A ランク**: 公式資料・信頼性の高いニュースソース → 「観測済み」ラベル使用可
- **B ランク**: 一般ニュース・ブログ → 「観測済み（参考）」ラベル
- **C ランク以下**: 信頼性不十分 → 根拠として不使用

### 2.3 アウトカム未観測ケースの扱い
- アウトカムが不明/未観測の事例は、シナリオの根拠ランクを1段階下げる
- 「仮説」ラベルを必ず付与

## 3. 適用条件と非適用条件

### 3.1 適用条件（このサービスが機能する前提）
- ユーザーが自分の状況を言語化できる（最低200文字以上のナラティブ）
- 変化・意思決定に関するテーマである
- ユーザーが自分で動く意思がある

### 3.2 非適用条件（このサービスが機能しない条件）
- 危機的状況（安全設計で遮断）
- 医療・法律・金融の専門判断が必要な場合
- ナラティブが極端に短い/抽象的で構造化不能な場合
- DBに類似事例が3件未満の場合（レポート生成を中止し、返金対応）

## 4. レポートへの根拠提示ルール

### 4.1 各シナリオに必須の情報
- 根拠ラベル（観測済み/仮説）
- 参照事例数（「N件の類似事例に基づく」）
- 前提条件（「この事例は〇〇の条件下で起きた」）
- 注意書き（「個別の状況により異なります」）

### 4.2 禁止事項
- 確率・パーセンテージの提示
- 「〇〇になります」等の断定的予測
- アウトカム未観測事例を「観測済み」として提示
- 個人が特定できる事例情報の提示（匿名化必須）

## 5. 匿名化ルール
- 個人名 → 「Aさん」「ある経営者」等に置換
- 企業名 → 公知の情報のみ使用可。非公知の場合は「同業界のB社」等
- 時期 → 「2010年代後半」等、幅を持たせる
- 具体的な金額 → 「数千万円規模」等に丸める
