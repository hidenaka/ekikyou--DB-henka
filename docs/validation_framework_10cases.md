## 1. 目的

ココナラMVP出品前に、自作ペルソナ10ケースでサービス全工程をドライランし、以下を検証する:

1. **フロー**: SOP通りに全工程が完走するか
2. **品質**: レポートが商品定義・禁止事項を満たしているか
3. **工数**: 実質時給が採算ラインを超えるか
4. **安全**: 危機ケースの検知・対応が正しく機能するか

## 4. 実施手順

### Step 1: ペルソナ作成（1日）

#### やること
- 10人分のペルソナシートを作成する
- 各ペルソナにナラティブ（200-500字）を用意する
- V06は意図的に短文（100字未満）にする
- V09, V10のナラティブは`safety_design.md`のキーワードカテゴリを参照して慎重に作成する

#### ペルソナシートの項目
- 名前（仮名）、年齢、性別
- 職業・家族構成
- 違和感の内容
- 制約条件
- リスク許容度
- 想定される構造化抽出結果（正解データ）
- ナラティブテキスト

#### 成果物
- `validation/personas/V01_persona.md` 〜 `validation/personas/V10_persona.md`

### Step 2: ドライラン（3-5日）

#### やること
- 1日2-3ケースずつ実施する
- 全工程をSOP通りに実行する（ココナラのUIは使わず、テキストベースでシミュレーション）
- 各ステップの所要時間をタイマーで計測する
- 全LLM入出力を記録保存する

#### 実行フロー（1ケースあたり）

```
1. ナラティブテキストを入力として用意する
2. Safety Filter実行（L1キーワード → L2 LLM分類）
   - CRITICAL/HIGH → 遮断テンプレート送信で終了（V10）
   - MEDIUM → 注意フラグ付きで継続（V09）
   - LOW → 通常処理
3. LLM構造化抽出を実行する
4. confidence < 0.6 の項目を確認する
   - あり → フォールバック質問を生成 → ペルソナとして回答 → 再抽出
   - なし → 次へ
5. 要約を生成する
6. ペルソナ視点で要約を評価する（合っている/違う/補足）
7. レポートを生成する
8. Gate 1-5の品質チェックを実行する
9. 人間レビューを実施する
10. 納品メッセージを作成する
```

#### 記録先
- `validation/runs/V01_run.md` 〜 `validation/runs/V10_run.md`
- `validation/llm_logs/` に全LLM入出力を保存する

### Step 3: 集計・分析（1日）

#### やること
- 全指標の集計
- Pass/Fail判定（セクション5参照）
- 改善点の洗い出し

#### 成果物
- `validation/summary.md`（集計結果と判定）

### Step 4: 改善（1-2日）

#### やること
- 不合格指標の原因分析
- テンプレート/プロンプト/SOPの修正
- 必要に応じて2-3ケースで再テスト

#### 成果物
- 修正されたテンプレート/プロンプト
- `validation/retest/` に再テスト結果を保存する

## 5. Pass/Fail基準

### 5.1 Go（ココナラ出品OK）

以下を**全て**満たす:

| # | 基準 | 閾値 |
|---|------|------|
| G1 | 禁止語ゼロ | 10/10ケースで達成 |
| G2 | Safety偽陰性 | 0件 |
| G3 | 構造化抽出confidence平均 | >= 0.65（V01-V08の全項目平均） |
| G4 | 要約合意率 | >= 80%（初回合意 8/10件以上） |
| G5 | フル版実質時給 | >= ¥7,000 |
| G6 | レポート品質ゲート通過率 | >= 90%（Gate 1-5全通過が9/10件以上） |
| G7 | フロー完走率 | 100%（V01-V09の9件全てが3往復以内で完走） |

### 5.2 Conditional Go（条件付き出品OK）

- 上記G1-G7のうち**1-2項目が未達**
- ただしG2（Safety偽陰性0件）は必須。これが未達なら即No Go
- 未達項目の改善計画を文書化した上で出品する
- 出品後、最初の5件で未達項目を再検証する

### 5.3 No Go（出品不可）

以下のいずれかに該当:

- Safety偽陰性が1件以上
- 3項目以上が未達
- フル版実質時給が¥5,000未満
- フロー完走率が80%未満

## 10. 注意事項

### Safety ケース作成時の注意
- V09（MEDIUM）: 重度抑うつのキーワードを含むが、文脈上は「少し気分が落ちている」程度のナラティブを作成する。Layer 2 LLM分類でMEDIUMと判定されることを確認する
- V10（CRITICAL）: 自傷・自殺に関するキーワードを含むナラティブを作成する。Layer 1キーワード検知 + Layer 2 LLM分類でCRITICALと判定されることを確認する
- Safetyケースのナラティブは検証目的でのみ使用し、検証完了後はllm_logsから削除する

### 自己評価バイアスの軽減
- 品質評価（B4, B6, B7, B8）は自己採点のため、バイアスが入りやすい
- 対策: 各ケースの評価前に「このレポートを受け取ったペルソナは本当にこれで動けるか」を基準にする
- Conditional Go以上の判定が出た場合、ココナラ出品後の最初の5件で実ユーザーからのフィードバックを再検証する

### 工数計測のルール
- タイマーは「実作業時間」のみ計測する（ユーザー返信待ち時間は除外）
- LLM APIの応答待ち時間は含める（実運用でも発生するため）
- 記録・集計の時間は含めない（検証用の作業であり、実運用では発生しない）
