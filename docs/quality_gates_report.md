# レポート品質ゲート仕様書

**作成日**: 2026-03-09
**ステータス**: 確定（Phase 0）
**対象**: 「人生の現在地」意思決定支援サービスのレポート品質管理
**前提ドキュメント**: `docs/product_definition_v3.md`, `docs/safety_design.md`, `docs/question_flow_v2.md`

---

## 設計原則

- 易経は内部フレームワーク。ユーザーに見せない
- 確率・予測・断定・占い用語は一切使わない
- ココナラで手動納品。人間レビューを経てから納品する
- 自動チェック（Gate 1-3）で機械的に弾けるものは弾き、人間チェック（Gate 4-5）でトーンと安全性を担保する

---

## レビューフロー

```
レポート生成
    │
    ▼
Gate 1: 禁止語チェック（自動）
    │ 不合格 → 自動再生成
    ▼
Gate 2: 構造チェック（自動）
    │ 不合格 → 自動再生成
    ▼
Gate 3: 根拠チェック（自動 + 人間）
    │ 不合格 → 事例の差し替え or 根拠ラベルの修正
    ▼
Gate 4: トーンチェック（人間）
    │ 不合格 → 人間が手動修正
    ▼
Gate 5: 安全チェック（人間）
    │ 不合格 → 人間が手動修正
    ▼
納品
```

## Gate 1: 禁止語チェック（自動）

レポートテキスト中に以下の語句・パターンが含まれていないことを確認する。

### 禁止語一覧

| カテゴリ | 禁止語・パターン | 重要度 |
|---------|----------------|--------|
| 断定的予測 | 「〜になります」「〜でしょう」「〜に違いない」「〜するはずです」 | CRITICAL |
| 確率表現 | 「〇%」「確率」「可能性が高い/低い」「的中率」「成功率」 | CRITICAL |
| 易経用語 | 「卦」「爻」「八卦」「乾」「坤」「震」「巽」「坎」「離」「艮」「兌」「陰」「陽」「易経」「変化の書」 | CRITICAL |
| 占い用語 | 「運勢」「運命」「宿命」「相性」「開運」「厄」「吉凶」 | CRITICAL |
| 推奨表現 | 「おすすめ」「推奨」「最適」「ベスト」「正解」「間違い」 | HIGH |
| 義務表現 | 「〜すべき」「〜しなければならない」「〜する必要がある」 | MEDIUM |
| タイプ分け | 「〇〇タイプ」「〇〇型」「あなたは〇〇な人」 | MEDIUM |
| 技術用語 | 「current_state」「energy_direction」「intended_action」「trigger_nature」「phase_stage」「constraints」「risk_tolerance」「choice_set」「confidence」「extraction」 | HIGH |

### 判定ルール

- **CRITICALが1つでもあれば不合格**。自動再生成に回す
- HIGH以下は修正指示付きで通過可。人間レビュー時に最終判断
- 義務表現の例外: 実験プラン（セクション6）内の判断基準として使用する場合は除外

### 実装上の注意

- 「〜になります」は文脈判断が必要。丁寧語としての「なります」（例: 「レポートの構成は以下のようになります」）は除外
- 「可能性が高い/低い」は部分一致で検出。「可能性」単独はHIGHとして検出
- 正規表現パターン: `\d+%` で確率表現を検出
- 技術用語はスネークケース・キャメルケース両方で検出

## Gate 2: 構造チェック（自動）

レポートのセクション構成と分量が仕様を満たしているかを確認する。

### フル版（6セクション）

| チェック項目 | 基準 | 判定 |
|-------------|------|------|
| セクション1: 現在地の構造化 | 存在すること | MUST |
| セクション2: このまま進んだ場合のシナリオ | 存在すること | MUST |
| セクション3: 別ルート 2-3本 | 存在すること | MUST |
| セクション4: 各ルートの比較表 | 存在すること | MUST |
| セクション5: 今日の1アクション | 存在すること | MUST |
| セクション6: 30日/90日実験プラン | 存在すること | MUST |
| 総文字数 | 3,000-5,000字 | SHOULD |
| 事例引用数（セクション2） | 3件以上 | MUST |
| 事例引用数（セクション3） | 各ルート2件以上 | MUST |
| 根拠ラベル | 全事例に「観測済み」or「仮説」ラベル | MUST |
| 比較表（セクション4） | 表形式（マークダウンテーブル）の比較あり | MUST |
| 今日の1アクション（セクション5） | 所要時間付きの具体的行動提案あり | MUST |
| 実験プラン（セクション6） | 30日プラン + 90日プランの両方あり | MUST |

### 軽量版（3セクション）

| チェック項目 | 基準 | 判定 |
|-------------|------|------|
| セクション1: 現在地の構造化 | 存在すること | MUST |
| セクション4: 各ルートの比較表 | 存在すること | MUST |
| セクション5: 今日の1アクション | 存在すること | MUST |
| 総文字数 | 1,000-2,000字 | SHOULD |
| 比較表（セクション4） | 表形式の比較あり | MUST |
| 今日の1アクション（セクション5） | 所要時間付きの具体的行動提案あり | MUST |

### 判定ルール

- **MUSTが1つでも欠ければ不合格**。自動再生成に回す
- SHOULDは警告のみ。人間レビュー時に判断

### セクション検出方法

- マークダウンの見出し（`##` or `###`）でセクションを検出
- セクション番号または名称のいずれかで一致判定
- 見出しがない場合でも、内容から推定可能であれば通過（ただし警告を出す）

## Gate 3: 根拠チェック（自動 + 人間）

事例の引用が品質基準を満たしているかを確認する。

### 自動チェック項目

| チェック項目 | 基準 | 判定 |
|-------------|------|------|
| 類似度スコア | 全事例の類似度 >= 0.50 | MUST |
| 高類似事例率 | 類似度 >= 0.70 の事例が全体の50%以上 | SHOULD |
| 事例数不足表示 | 3件未満のシナリオに「事例数不足」ラベルあり | MUST |
| 根拠ラベル整合性 | 「観測済み」事例にアウトカム記述あり | MUST |
| 「仮説」ラベル整合性 | 「仮説」事例に推論根拠の記述あり | MUST |

### 人間チェック項目

| チェック項目 | 基準 |
|-------------|------|
| 匿名化 | 個人名・企業名が適切に匿名化されている（公知の歴史的事例を除く） |
| 事例の妥当性 | 引用事例がユーザーの状況と本当に類似しているか |
| アウトカム正確性 | 「観測済み」事例のアウトカムが事実と一致しているか |
| 多様性 | 引用事例が偏っていない（全て成功事例、全て失敗事例になっていない） |

### 判定ルール

- 自動チェックのMUSTが1つでも不合格 → 事例の差し替え or 根拠ラベルの修正
- 人間チェックで問題あり → 手動で事例を差し替え

## Gate 4: トーンチェック（人間）

レポートのトーンがサービスの設計意図に合致しているかを確認する。

### チェック項目

| チェック項目 | 基準 | 確認のポイント |
|-------------|------|-------------|
| ユーザーの言葉の引用 | セクション1にユーザー自身の言葉が最低2箇所引用されている | 「」付きの直接引用があるか |
| 中立性 | 特定のルートを推していない | セクション3-4で特定ルートに肯定的な形容詞が偏っていないか |
| 共感性 | 分析的すぎず、ユーザーの違和感に寄り添っている | セクション1で感情的な側面にも触れているか |
| 実験トーン | 「失敗」ではなく「実験」のフレーミング | 「もし上手くいかなくても」等のネガティブ表現が「実験として試す」に置き換わっているか |
| 行動の具体性 | セクション5の行動が5W1Hで特定可能 | いつ・どこで・何を・どのくらいの時間で、が明記されているか |
| 過剰な楽観/悲観 | バランスが取れている | ポジティブ・ネガティブ両面に言及しているか |
| 押しつけのなさ | 選択の主体がユーザーであることが明確 | 「あなたが決める」「試してみて、判断する」のトーンになっているか |

### 判定ルール

- 上記のうち2項目以上で問題あり → 不合格。人間が手動修正
- 1項目のみ → 修正推奨。軽微であれば通過可

## Gate 5: 安全チェック（人間）

レポートに専門領域への踏み込みや過度な方向づけがないかを確認する。

### チェック項目

| チェック項目 | 基準 | 具体例 |
|-------------|------|--------|
| 医療助言の混入 | 医療・心理の専門的助言が含まれていない | 「うつの可能性があります」「カウンセリングを受けてください」等 |
| 法律助言の混入 | 法的助言が含まれていない | 「労基署に通報すべき」「慰謝料を請求できます」等 |
| 金融助言の混入 | 投資・資産運用の助言が含まれていない | 「この投資は有利です」「貯蓄を増やしましょう」等 |
| 過度な方向づけ | 特定の行動を強く促していない | 「絶対に転職すべき」「今の会社を辞めてはいけない」等 |
| ディスクレーマー | 注意書きが含まれている | レポート末尾に免責事項あり |
| スコープ逸脱 | サービスが扱わないテーマに踏み込んでいない | `docs/safety_design.md` セクション5のスコープ外テーマ |

### ディスクレーマーの必須文言

レポート末尾に以下の趣旨の文言が含まれていること:

```
当レポートは意思決定の整理をお手伝いする参考資料です。
提示するシナリオは過去の類似事例に基づくパターン提示であり、
将来の結果を保証するものではありません。
医療・心理・法律・金融の専門的な助言は含まれておりません。
最終的な判断はご自身で行ってください。
```

### 判定ルール

- 上記のうち1項目でも問題あり → 不合格。人間が手動修正
- ディスクレーマーの欠落 → 不合格

## チェック自動化スクリプト仕様

### 基本情報

- スクリプト名: `scripts/check_report_quality.py`
- 入力: レポートテキスト（マークダウン形式）
- 出力: Gate 1-3の自動チェック結果（JSON） + Gate 4-5の人間チェック用チェックリスト

### 入力方式

```bash
# ファイルから
python3 scripts/check_report_quality.py --file report.md --plan-type full

# 標準入力から
cat report.md | python3 scripts/check_report_quality.py --plan-type light
```

## 運用ルール

### 自動再生成の上限

- Gate 1/2での自動再生成は最大3回まで
- 3回連続で不合格の場合、人間に通知して手動対応

### 品質ゲートのバージョン管理

- 禁止語リストの追加・削除は本ドキュメントを更新し、スクリプトに反映
- 月次でレビューし、偽陽性/見逃しの傾向を確認して更新

### ログ記録

- 全レポートのGate 1-5の結果をログに保存
- 不合格理由の集計を月次で実施し、レポート生成プロンプトの改善に使用
