# 入力マッピング設計 v2.0 -- テキスト相談 → 卦・爻マッピング

**作成日**: 2026-02-25
**前提**: Phase3同型性検証(ANALOGOUS)完了、Codex批評統合済み、structured_question_flow_design v1.0既存

---

## 1. 問題の構造

ユーザーが自然言語テキストで状況を相談する。システムはそのテキストから以下を決定する必要がある:

| 要素 | 決定内容 | 空間サイズ |
|------|----------|-----------|
| **本卦** | 下卦(内面) + 上卦(環境) | 8 x 8 = 64通り |
| **動爻** | 変化が起きているフェーズ | 6通り |
| **之卦** | 動爻反転後の卦 | 本卦と動爻から自動算出 |

合計: 64 x 6 = 384パターンへの分類問題。

### 前回の失敗から得た教訓

Codex批評の核心指摘:
1. **NLP分類層が致命的ボトルネック** -- テキスト→八卦の直接マッピングは検証不可能
2. **事後タグ付けDBで事前診断は作れない** -- 13,060件は結果を知った上でタグ付けされたデータ
3. **入力マッピング誤差は後段全体を汚染する** -- ここが不安定だと確率表も事例検索も無意味

## 2. DB実態から得られた経験的マッピング規則

13,060件の分析で、**人間可読ラベルと八卦タグの間に強い経験的対応関係**が存在することが判明した。これは設計の最大の資産である。

## 3. 設計案

### 設計案A: LLM意味解析 + 構造化中間表現 + 決定論的マッピング（推奨）

```
[ユーザーのテキスト相談]
         |
         v
  ┌─────────────────────────┐
  │  LLM 意味解析エンジン    │  Claude/GPT に八卦知識を注入した
  │  (プロンプト + few-shot) │  プロンプトで構造化中間表現を抽出
  └───────────┬─────────────┘
              |
              v
  ┌─────────────────────────┐
  │  構造化中間表現 (JSON)    │  ← これが「確定ゲート」
  │  - current_state (6択)   │
  │  - intended_action (8択) │
  │  - trigger_nature (6択)  │
  │  - phase_stage (6択)     │
  │  - domain (5択)          │
  │  - energy_direction (2択)│
  │  - movement_tendency (2択)│
  └───────────┬─────────────┘
              |
              v
  ┌─────────────────────────┐
  │  確率的マッピングエンジン  │  DBの経験的対応関係に基づく
  │  (条件付き確率テーブル)   │  確率加重マッピング
  └───────────┬─────────────┘
              |
              v
  ┌─────────────────────────┐
  │  本卦 + 動爻 + 之卦      │  → 事例検索 → フィードバック
  └─────────────────────────┘
```

#### 処理詳細

**Step 1: LLMによる構造化抽出**

LLMに以下のプロンプトを送る:

```
あなたは変化の構造分析者です。ユーザーの状況記述から、以下の7項目を抽出してJSON形式で返してください。

解釈に迷う場合は confidence を下げ、複数候補を出してください。

{
  "current_state": {
    "primary": "安定・平和|成長痛|停滞・閉塞|どん底・危機|絶頂・慢心|混乱・カオス",
    "confidence": 0.0-1.0,
    "reasoning": "判断根拠を1文で"
  },
  "intended_action": {
    "primary": "攻める・挑戦|守る・維持|耐える・潜伏|対話・融合|刷新・破壊|捨てる・撤退|逃げる・放置|分散・スピンオフ",
    "confidence": 0.0-1.0,
    "reasoning": ""
  },
  "trigger_nature": {
    "primary": "外部ショック|内部崩壊|意図的決断|偶発・出会い|内部矛盾・自壊|自然推移",
    "confidence": 0.0-1.0,
    "reasoning": ""
  },
  "phase_stage": {
    "primary": "潜伏・発芽|出現・成長|危機・転換|選択・準備|最盛・実行|過剰・衰退",
    "confidence": 0.0-1.0,
    "reasoning": ""
  },
  "domain": "company|individual|family|country|other",
  "energy_direction": "expanding|contracting",
  "movement_tendency": "active|still"
}
```

few-shotとして、DBから代表事例の `story_summary` → 構造化表現ペアを5-10件添付する。

**Step 2: 構造化中間表現からの確率的マッピング**

DBの経験的対応テーブルを使い、各八卦への確率を算出する:

```python
# 下卦の決定（内的状態 = current_state が最大の決定因子）
P(下卦=X) = w1 * P(before_hex=X | current_state)
           + w2 * P(before_hex=X | energy_direction, movement_tendency)

# 上卦の決定（環境・行動方向 = trigger + intended_action が決定因子）
P(上卦=X) = w3 * P(action_hex=X | intended_action)
           + w4 * P(trigger_hex=X | trigger_nature)
           + w5 * P(上卦=X | energy_direction_external, movement_external)

# 爻位の決定（フェーズから直接マッピング）
yao = phase_stage → {潜伏:1, 出現:2, 危機:3, 選択:4, 最盛:5, 過剰:6}
```

重み `w1-w5` はDBのCramer's Vから初期設定し、将来のフィードバックで調整可能:
- `w1 = 0.75` (current_state→下卦は対応が強い)
- `w2 = 0.25` (エネルギー方向は補助)
- `w3 = 0.50` (行動→上卦)
- `w4 = 0.35` (トリガー→上卦)
- `w5 = 0.15` (エネルギー方向は補助)

**Step 3: 確信度による分岐**

```
if max(P(下卦)) > 0.60 and max(P(上卦)) > 0.50:
    → 自動決定。結果をユーザーに提示し確認を求める
elif max(P(下卦)) > 0.40 or max(P(上卦)) > 0.35:
    → 上位2候補を提示し、ユーザーに選択してもらう
else:
    → フォールバック: 構造化質問（設計案B）に切り替え
```

#### メリット
1. **ユーザー体験が自然** -- テキストを投げるだけで卦が返る。質問責めにしない
2. **DBの13,060件の知識を最大活用** -- 確率テーブルは実データから構築
3. **LLMの強みを正しく使う** -- LLMにやらせるのは「テキスト→6カテゴリラベル」という得意領域。八卦への直接マッピングという困難なタスクはさせない
4. **確信度が低い時は人間に委ねる** -- 曖昧入力でも破綻しない
5. **検証可能** -- 中間表現が構造化されているため、どこで間違えたか追跡できる
6. **之卦は自動算出** -- 本卦と動爻が決まれば動爻を反転させるだけ

#### デメリット
1. **LLM APIコストが毎リクエスト発生** -- 1回あたり約$0.01-0.03
2. **LLMの構造化抽出精度に依存** -- 曖昧な相談文では current_state の判定がブレる
3. **確率テーブルの重みチューニングが必要** -- 初期値はDBから出せるが最適化は別途必要
4. **レスポンスタイム** -- LLM呼び出しで2-5秒

#### 実現可能性: **78/100**

最もバランスが良い。LLMは「テキスト→構造化ラベル」という得意なタスクのみ担当し、八卦マッピングは経験的確率テーブルが担う。致命的な単一障害点がない。

### 設計案B: 完全構造化入力（Codex推奨路線の発展版）

```
[ユーザー]
    |
    v
┌──────────────────────────────┐
│  Step 1: テーマ設定 (2問)     │
│  - 視点（自分/組織/家族/他者） │
│  - 領域（キャリア/関係/事業等）│
└──────────┬───────────────────┘
           |
           v
┌──────────────────────────────┐
│  Step 2: 内的状況 = 下卦 (3問) │
│  Q3: エネルギー方向 (陽/陰)   │
│  Q4: 動静 (動/静)            │
│  Q5: 象徴選択 (8択から1つ)    │
└──────────┬───────────────────┘
           |
           v
┌──────────────────────────────┐
│  Step 3: 外的状況 = 上卦 (3問) │
│  Q6: エネルギー方向 (陽/陰)   │
│  Q7: 動静 (動/静)            │
│  Q8: 象徴選択 (8択から1つ)    │
└──────────┬───────────────────┘
           |
           v
┌──────────────────────────────┐
│  Step 4: フェーズ = 爻位 (1問) │
│  Q9: 進展段階 (6択)           │
└──────────┬───────────────────┘
           |
           v
  本卦(64) + 動爻(6) → 之卦 → 事例検索
```

これは既存の `structured_question_flow_design.md` (v1.0) をそのまま活用する。

#### 決定論的マッピング

- Q3(2択) x Q4(2択) x Q5(最大2択に絞られる) = 8パターン → 下卦確定
- Q6(2択) x Q7(2択) x Q8(最大2択に絞られる) = 8パターン → 上卦確定
- Q9(6択) → 爻位確定
- 合計 **9問で384パターンに一意決定**

#### メリット
1. **100%決定論的** -- 同じ回答は同じ卦に必ずなる。再現性完璧
2. **LLM不要** -- APIコストゼロ、レスポンス即時
3. **マッピング誤差ゼロ** -- ユーザーが自分で選んだ結果だから、「マッピングが間違った」という問題が構造的に発生しない
4. **Codex推奨に完全準拠** -- 「NLPを捨てて構造化入力」の要件を満たす
5. **検証が容易** -- 回答→卦の対応表を公開でき、ブラックボックスがない

#### デメリット
1. **ユーザー体験が硬い** -- 9問の質問に答える必要があり、「相談している」感覚が薄い
2. **ユーザーの自己認知バイアス** -- 「自分は積極的だ」と思い込んでいる人が実は停滞状態、といったケースを検出できない
3. **Q5/Q8の象徴選択が難しい** -- 「雷のような突発的変化」か「炎のような情熱」か、一般ユーザーには判断困難
4. **テキスト相談という要件を満たさない** -- 「テキストで状況を投げる」という本来の要件から逸脱している
5. **DBの13,060件の知識を活用していない** -- マッピングにDBデータを使わない

#### 実現可能性: **92/100**

技術的には最もシンプルで確実。ただし「テキスト相談→フィードバック」という元の要件との乖離が大きい。

### 設計案C: LLM直接マッピング（八卦知識注入型）

```
[ユーザーのテキスト相談]
         |
         v
┌────────────────────────────────────┐
│  LLM (Claude/GPT)                   │
│  + 八卦意味体系のシステムプロンプト   │
│  + 64卦の全解説をRAG/context注入     │
│  + DB事例10-20件をfew-shot           │
│                                      │
│  出力:                               │
│  {                                   │
│    "hexagram_number": 1-64,          │
│    "yao": 1-6,                       │
│    "reasoning": "..."                │
│  }                                   │
└───────────┬────────────────────────┘
            |
            v
     本卦 + 動爻 → 之卦 → 事例検索
```

#### LLMへの知識注入内容

1. **八卦の意味体系** (8卦の定義 + 象意 + 対比構造)
2. **64卦の解説** (`iching_texts_ctext_legge_ja.json` から卦辞・象伝・爻辞)
3. **DB事例のfew-shot** (テキスト→卦のペアを10-20件)
4. **マッピング基準の明示的記述**:
   - 下卦=相談者の内的状態
   - 上卦=外的環境・相手の性質
   - 爻位=変化の進展段階

#### 実験的検証（やるべきテスト）

```python
# テスト: 同一テキストを10回投げて一貫性を測定
consistency_test = {
    "同一テキスト10回": "同じ卦が返る割合",
    "合格基準": ">= 80% (10回中8回同じ)",
    "temperature": 0.0  # 必須
}

# テスト: DB事例のstory_summaryを投げて既存タグとの一致を測定
accuracy_test = {
    "DB事例100件のstory_summary → LLM → 卦": "",
    "比較対象": "hexagram_number + yao (既存タグ)",
    "合格基準": "hexagram一致 >= 40%, 上下卦個別一致 >= 60%"
}
```

#### LLMが八卦を理解しているか？

2026年のLLM (Claude Opus 4.6, GPT-5.2) は易経の基本的な意味体系を訓練データから理解している。ただし:

- **強い点**: 八卦の象意（乾=創造/リーダーシップ、坎=困難/水、等）の基本理解は正確
- **弱い点**: 64卦の微妙な区別（例: 水天需 vs 水地比）は不安定。爻辞レベルの判断は更に不安定
- **補える点**: few-shotとコンテキスト注入で精度は大幅改善可能
- **補えない点**: temperature=0でも、境界ケースでの判断は揺れる

#### メリット
1. **ユーザー体験が最も自然** -- テキストを投げるだけ
2. **実装が最もシンプル** -- プロンプトエンジニアリングのみ
3. **LLMの推論能力を最大活用** -- 文脈理解+八卦マッピングを一気通貫
4. **DB事例をfew-shotとして直接活用** -- 学習データとの接続が簡単

#### デメリット
1. **一貫性の保証が困難** -- 同じテキストでも卦が揺れるリスク（最大の問題）
2. **判断過程がブラックボックス** -- なぜその卦を選んだか、追跡困難
3. **LLMの八卦理解は「浅い」** -- 表面的な象意マッチングに陥りやすい。64卦の構造的関係（錯卦・綜卦・互卦）を考慮した判断は期待できない
4. **コスト高** -- 64卦全解説をcontextに入れると毎回大量トークン消費
5. **検証が困難** -- LLMの内部判断を分解できないため、「どこが間違えたか」の診断ができない
6. **Codex批評の致命的指摘に対応していない** -- 「NLP分類層がボトルネック」の問題をLLMに置き換えただけで構造的に解決していない

#### 実現可能性: **45/100**

実装は容易だが、一貫性と検証可能性の問題が致命的。プロダクトとしての信頼性を担保できない。

## 5. 最終推奨: 設計案A（LLM+構造化中間表現）をベースに、設計案Bをフォールバックとする二層構造

### 推奨アーキテクチャ

```
[ユーザーのテキスト相談]
         |
         v
 ┌──────────────────────┐
 │  Layer 1: LLM抽出     │  テキスト → 構造化中間表現(JSON)
 │  (Claude API)          │  confidence付き7項目
 └──────────┬───────────┘
            |
            v
      ┌──── confidence判定 ────┐
      |                        |
  [高確信]                  [低確信]
  confidence               confidence
  >= 0.6                   < 0.6
      |                        |
      v                        v
 ┌──────────┐          ┌──────────────┐
 │ 確率テーブル│          │ Layer 2:      │
 │ マッピング  │          │ 構造化質問     │
 │ (DB由来)   │          │ (設計案B)      │
 └─────┬────┘          │ 足りない項目    │
       |                │ のみを質問      │
       v                └──────┬───────┘
 ┌──────────┐                  |
 │ 候補提示   │                  v
 │ + 確認     │           ┌──────────┐
 │ (上位2卦)  │           │ 決定論的   │
 └─────┬────┘           │ マッピング  │
       |                 └──────┬───┘
       v                        |
  ┌────────────────────────────┘
  |
  v
 本卦 + 動爻 → 之卦算出 → DB事例検索 → フィードバック生成
```

### 設計のポイント

**1. LLMは「翻訳者」であって「判断者」ではない**

LLMの仕事は「自然言語テキスト → 6つのカテゴリラベル」への翻訳のみ。八卦への変換はDBの経験的確率テーブルが行う。これによりLLMの八卦理解の浅さを回避し、DBの13,060件の知識を最大活用する。

**2. confidence閾値で自動/対話を切り替える**

- 全項目 confidence >= 0.7 → 完全自動マッピング、候補2つ提示
- いずれか confidence < 0.6 → その項目のみ構造化質問で確認
- 複数項目 confidence < 0.4 → 全体的に構造化質問モードに切り替え

**3. ユーザー確認を必ず挟む**

卦の決定は最終的にユーザーが「これは自分の状況に合っている」と感じることが重要。自動決定でも「あなたの状況はこのように読み取りました。合っていますか？」の確認ステップを入れる。

### 之卦の算出

之卦は本卦と動爻から機械的に算出される:
- 本卦の6つの爻（各爻は陰or陽）のうち、動爻の位置の爻を反転
- 反転後の上卦・下卦の組み合わせから64卦番号を算出

```python
# 八卦 → 3ビット表現
TRIGRAM_BITS = {
    "乾": (1,1,1), "兌": (1,1,0), "離": (1,0,1), "震": (1,0,0),
    "巽": (0,1,1), "坎": (0,1,0), "艮": (0,0,1), "坤": (0,0,0)
}

def get_zhi_gua(lower_trigram, upper_trigram, yao_position):
    """動爻を反転して之卦を求める"""
    bits = list(TRIGRAM_BITS[lower_trigram]) + list(TRIGRAM_BITS[upper_trigram])
    # yao_position: 1-6 (下から)
    idx = yao_position - 1
    bits[idx] = 1 - bits[idx]  # 反転
    new_lower = tuple(bits[0:3])
    new_upper = tuple(bits[3:6])
    # ビット→八卦の逆引き
    BITS_TRIGRAM = {v: k for k, v in TRIGRAM_BITS.items()}
    return BITS_TRIGRAM[new_lower], BITS_TRIGRAM[new_upper]
```

### 信頼性の担保

| 担保方法 | 詳細 |
|----------|------|
| **一貫性テスト** | 同一テキスト10回→構造化中間表現の一致率>=90%を要求。temperature=0必須 |
| **対DB検証** | DB事例200件のstory_summary→構造化抽出→確率マッピング→既存hexagram_numberとの比較。上下卦個別一致>=55%を合格基準 |
| **ユーザー確認** | 自動決定でも必ず「この読み取りは合っていますか？」を挟む |
| **フィードバックループ** | ユーザーの「合っている/違う」を蓄積し、確率テーブルの重みを更新 |
| **監査ログ** | 全リクエストの中間表現・確率・最終決定を記録し、事後分析可能にする |

## 6. 実装ロードマップ

| フェーズ | 内容 | 期間 | 判断基準 |
|----------|------|------|----------|
| **0. 検証** | DB事例100件でLLM構造化抽出の精度テスト | 3日 | current_state一致>=70%, action_type一致>=60% |
| **1. 確率テーブル** | DBから条件付き確率テーブルを構築 | 2日 | テーブル完成+単体テスト |
| **2. プロトタイプ** | テキスト→構造化→確率→卦のパイプライン | 5日 | E2Eで動作 |
| **3. フォールバック** | confidence低時の構造化質問切替 | 3日 | 低確信時にUIが切り替わる |
| **4. 事例検索** | 決定された卦+条件でDB検索 | 3日 | 類似事例top-5が返る |
| **5. 統合テスト** | 50人テストユーザーでの妥当性評価 | 2週間 | 「卦が状況に合っている」>=60% |

**フェーズ0で不合格の場合**: 設計案Bに切り替え。LLMは事例検索と解説生成のみに使用。

## 7. Codex批評への回答

| Codex指摘 | 本設計での対応 |
|-----------|---------------|
| NLPを捨てて構造化入力 | LLMの出力を構造化中間表現に限定。NLP→八卦の直接マッピングは行わない |
| 入力マッピング未検証 | フェーズ0で200件の検証テストを実施。不合格なら設計案Bにフォールバック |
| 事後タグDBで事前診断は不可 | 「診断」ではなく「状況マッピング+類似事例提示」と位置付け |
| データリーク | 構造化中間表現はbefore_state/action_typeのみ。outcome/after_stateは入力に使わない |
| 過信出力の防止 | 確率分布+事例件数(n)を必ず併記。断定口調禁止 |
| 領域差の分離 | domain(company/individual/等)を中間表現に含め、確率テーブルを領域別に分離可能 |
